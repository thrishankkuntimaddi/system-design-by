API Design — complete reference

APIs are the contracts between software components. This guide covers every layer — from transport protocols up to security patterns, versioning, and real-world design decisions.

What is an API?

Application Programming Interface. A documented contract that defines what requests can be made, how to make them, and what responses to expect. The interface between client and server.

API as contract

The API contract defines: valid requests, required parameters, authentication method, response shape, error codes. Break the contract = you break every consumer simultaneously.

Why API design matters

A bad API is forever. Once developers integrate your API, changing it breaks them. Good design reduces support burden, improves DX, enables scaling, and prevents security holes.

API consumers

Internal teams (microservices), external developers (third-party integrations), mobile apps, browsers, IoT devices, partner systems. Each has different needs and constraints.

API design quality spectrum

Bad API

Inconsistent naming. No versioning. 200 for all errors. No pagination. Returns entire object graph. Undocumented. No rate limiting. Changing behavior without notice.

OK API

Basic REST conventions. Some error codes correct. Some docs. No guarantees on stability. Versioning exists but inconsistent. Limited pagination support.

Great API

Consistent naming. Full OpenAPI docs. Proper status codes. Versioned with deprecation policy. Cursor pagination. Rate limit headers. Idempotency keys. Predictable evolution.

Design
4 key principles of great APIs

Every lasting API decision traces back to one of these four. Violate any one and you create technical debt that compounds over time.

1. Consistency

Same naming patterns everywhere. Same error shape everywhere. Same auth mechanism everywhere. If your user endpoints use snake_case and your product endpoints use camelCase — that's a bug in your design. Developers build mental models from patterns; break the pattern and you break their trust.

Consistent namingConsistent error shapesConsistent authConsistent response format

2. Simplicity

Focus on core use cases. Every extra parameter, every optional field, every implicit behavior is surface area that must be maintained forever. KISS — Keep It Simple. An API is easy to use when it's hard to misuse. Avoid "god endpoints" that do 15 different things based on magic flags.

One resource per endpointMinimal required paramsPredictable behaviorHard to misuse

3. Security

Authentication — who are you? Authorization — what can you do? Input validation — is this data safe? Rate limiting — are you abusing the system? Security must be designed in from the start, not bolted on later. Every endpoint should ask: "What's the worst thing someone could do here?"

AuthN + AuthZ firstValidate all inputRate limit everythingPrinciple of least privilege

4. Performance

Caching strategies — serve from cache when fresh. Pagination — never return unbounded lists. Minimize payload size — return only what's needed. Reduce round trips — batch related data. Compression — gzip response bodies. One slow endpoint kills perceived performance of your whole product.

Mandatory paginationHTTP cachingMinimize round tripsCompress responses

Design-first vs code-first

Design-first (recommended)

Write OpenAPI spec before writing code. Review the API contract with consumers. Catch design mistakes before implementation. Generate server stubs and client SDKs from spec.

Code-first

Write code, then generate docs. Faster to start, but design mistakes get baked into implementation. Hard to change once developers depend on it. Common in early startups.

Protocols
HTTP / HTTPS

The foundation of data exchange on the web. Every REST API runs on HTTP. Understanding request/response structure is non-negotiable.

HTTP (Hypertext Transfer Protocol) — a stateless, text-based application protocol. Client sends a request with method + URL + headers + body. Server responds with status code + headers + body. No memory of previous requests.
Request / response anatomy

HTTP request

GET /api/products/123 HTTP/1.1
Host: api.example.com
Authorization: Bearer tk9382nf...
Accept: application/json
Content-Type: application/json
Accept-Encoding: gzip
User-Agent: MyApp/2.0

HTTP response

HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=3600
ETag: "abc123"
X-RateLimit-Remaining: 99

{ "id": 123, "price": 19.99 }
HTTPS = HTTP + TLS/SSL
Benefits of HTTPS: data encrypted in transit, data integrity (no tampering), server authentication via certificates, SEO benefits, browser trust indicators (padlock). Required for any authentication to be secure.
Risks without HTTPS: man-in-the-middle attacks, data tampering by ISPs or routers, information theft, credential exposure, session hijacking. Browsers now warn on HTTP. Never ship a production API over plain HTTP.
Common headers

Content-Type

Tells receiver the format of the body. application/json, multipart/form-data, text/plain. Server must set on responses; client must set on requests with body.

Authorization

Carries credentials. Bearer <token> for JWT/OAuth, Basic <base64> for basic auth. Never in URL query params — logs and history capture URLs.

Cache-Control

Instructs caches how long to store the response. max-age=3600 = cache for 1 hour. no-store = never cache. no-cache = revalidate before using cached copy.

ETag

Entity tag for conditional requests. Server returns ETag with response. Client sends If-None-Match: "abc123". Server returns 304 Not Modified if unchanged — saves bandwidth.

Rate limit headers

X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset. Standardizing as RateLimit (IETF draft). Always include so clients can back off gracefully.

Idempotency-Key

Client sends a unique UUID. Server deduplicates repeated requests. Critical for payment APIs — prevents double charges if network retries trigger a second POST.

Transport layer
TCP vs UDP

These transport protocols determine how data physically moves between machines. Everything above them — HTTP, WebSockets, gRPC — runs on top of one or the other.

TCP — Transmission Control Protocol

Reliable but slower. Guarantees every packet arrives and arrives in order. Uses the 3-way handshake to establish connection.

3-way handshake
1. SYN client → server: "I want to connect"
2. SYN-ACK server → client: "OK, I'm ready"
3. ACK client → server: "Connection established!"
Packet lost? TCP detects timeout → retransmits packet. All data arrives, in order. No data loss — ever.
Banking APIsEmailHTTP/RESTWebSocketsgRPC

UDP — User Datagram Protocol

Fast but unreliable. Fire-and-forget. No handshake, no delivery guarantee, no ordering. Less overhead = lower latency. The application decides what to do with lost packets.

Packet lost? UDP says "NO RESENDING" — just keeps going with the next packet. A frame skip is acceptable; a 300ms stutter to retransmit is not.
Video callsOnline gamingLive streamingDNSHTTP/3 (QUIC)
Rule: If losing a packet hurts less than waiting for a retransmit — use UDP. If every byte must arrive — use TCP.

Protocols
WebSockets

Persistent, bidirectional connection between client and server. Replaces HTTP polling for real-time applications. One connection, unlimited messages in both directions.

WebSocket — a protocol that upgrades an HTTP connection into a persistent, full-duplex channel. Once established, both client and server can push data at any time without waiting for a request. Runs over TCP.
HTTP polling vs WebSockets

HTTP polling (old way)

Client asks: "Any new data?" → Server: "No." → Client asks again: "Any new data?" → Server: "No." → Client asks... This repeats constantly, wasting bandwidth and adding latency.

Increased latency. Wasted bandwidth on empty responses. Server resource overhead from repeated connections. Not truly real-time — always 1+ polling interval behind.

WebSocket (right way)

Client sends one HTTP handshake → connection upgrades to WebSocket → stays open. Server pushes data the instant it's available. Client can also send messages at any time.

Real-time data with minimal latency. Reduced bandwidth. Bidirectional communication over a single persistent connection.
WebSocket handshake
// Client sends HTTP upgrade request:
GET /ws HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZQ==

// Server responds 101 Switching Protocols
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
// Now full-duplex — ws:// or wss:// for TLS
WebSockets vs SSE

WebSockets (bidirectional)

Full-duplex — both sides send. Better for chat, gaming, collaborative editing. Needs custom reconnection logic. Browser API: new WebSocket(url).

SSE — Server-Sent Events (server→client only)

One-way — server pushes, client just receives. Simpler, automatic reconnection. Good for dashboards, live feeds. Uses standard HTTP — works through proxies. Browser API: new EventSource(url).

Use cases
Chat apps (WhatsApp, Slack)Live trading dashboardsMultiplayer gamesLive collaboration (Figma, Notion)Real-time notificationsLive sports scores

AMQP — Advanced Message Queuing Protocol

Enterprise messaging protocol. Producers publish messages to a broker; consumers pull from queues. Guaranteed delivery, async communication, service decoupling.

AMQP is a wire-level protocol for message-oriented middleware. Unlike HTTP (direct client-server), AMQP introduces a message broker in between. Producers don't talk to consumers directly — they talk to the broker.

Exchange types

Direct (1:1)

Message goes to exactly one queue. Like a direct phone call. Used when one consumer should handle each message. Routing key must match binding key exactly.

Fanout (1:N)

Message broadcast to all bound queues. Like a radio station. Used for notifications that every service needs to see. Routing key is ignored.

Topic (pattern-based)

Routes based on routing key pattern matching. order.*.placed matches order.uk.placed. Flexible routing logic for complex event-driven systems.

Key benefit: AMQP decouples producer from consumer. Producer doesn't need to know if consumers exist, how many there are, or if they're available. Messages wait in the queue until consumed. Dead letter queues capture failed messages for retry.

Protocols
gRPC — Google Remote Procedure Call

High-performance, binary protocol for server-to-server communication. Used heavily in microservices architectures. Up to 10x faster than JSON/REST for equivalent operations.

gRPC uses Protocol Buffers (protobuf) for serialization and HTTP/2 as the transport. Services are defined in .proto files — strongly typed contracts. Client stubs are auto-generated in any language (Go, Java, Python, Rust…).

Protocol Buffers (.proto)

Binary serialization format (not JSON). Schema-defined in .proto files. Smaller payload than JSON (~5–10x), faster to serialize/deserialize. Type-safe — schema mismatch is caught at compile time, not runtime.

// user.proto
syntax = "proto3";

message User {
  string id = 1;
  string name = 2;
  int32 age = 3;
}

service UserService {
  rpc GetUser (UserRequest) returns (User);
  rpc ListUsers (stream UserRequest) returns (stream User);
}

4 communication types

Unary — one request, one response (like REST).
Server streaming — one request, many responses (server pushes data as available).
Client streaming — many requests, one response (client uploads chunks).
Bidirectional streaming — both sides stream simultaneously. Use for real-time coordination.

Protocols
Choosing the right protocol

Choosing the wrong protocol creates a bottleneck. These 6 factors determine the right call. Most systems use multiple protocols for different needs.

Interaction pattern

Request-response → HTTP/REST
Real-time push → WebSockets
Async fire-and-forget → AMQP / Kafka
Service-to-service streaming → gRPC
Server → client only → SSE

Performance requirements

gRPC (binary, HTTP/2) is fastest for internal services. REST (text, HTTP/1.1) has more overhead. WebSockets are best for sustained real-time. AMQP is async — latency tolerance required. For ultra-low latency streaming: gRPC or WebSockets.

Client compatibility

Browsers → REST or GraphQL (gRPC needs proxy).
Mobile → REST or gRPC.
Internal services → gRPC.
IoT → MQTT or AMQP.
Third-party integrations → REST (widest tool support).

Payload characteristics

Large binary payloads → gRPC (protobuf). Small JSON → REST. Complex nested data → GraphQL. Streaming large datasets → gRPC server streaming. Sparse/sparse data → GraphQL (request only fields needed).

Security needs

Always use TLS/SSL encryption in transit. gRPC has TLS built-in via HTTP/2. HTTPS for REST. WSS (WebSocket Secure) for WebSockets. mTLS (mutual TLS) for zero-trust internal services.

Developer experience

REST has the best tooling (Postman, Swagger, curl, browser DevTools). GraphQL has Playground and introspection. gRPC requires proto toolchain setup. Consider team familiarity and hiring pool.

API design
RESTful APIs

Resource-based URL design, filtering, sorting, pagination, and versioning — the complete REST playbook. REST is the dominant style for public APIs.

REST lets different parts of a system communicate using standard HTTP methods and resource-based URLs. It has 6 constraints: stateless, uniform interface, client-server separation, layered system, cacheable, and code on demand (optional).
Resource modeling — URL patterns
Collection of resources:
GET   /api/v1/products
POST  /api/v1/products

Specific resource:
GET    /api/v1/products/:id
PATCH  /api/v1/products/:id
DELETE /api/v1/products/:id

Nested resource (relationship):
GET /api/v1/products/:id/reviews
POST /api/v1/products/:id/reviews

Actions (when needed — use nouns where possible):
POST /api/v1/orders/:id/cancel
POST /api/v1/users/:id/verify-email
Filtering, sorting, and field selection

Filtering — query parameters

Pass filter criteria in query string. Server applies them before returning results. Support multiple filters.

GET /api/v1/products?category=electronics&inStock=true&minPrice=10&maxPrice=500

Sorting

Sort key and direction as query params. Support multiple sort keys.

GET /api/v1/products?sort=price_asc
GET /api/v1/products?sort=-created_at  // prefix minus = desc
GET /api/v1/products?sort=category,price_asc

Field selection (sparse fieldsets)

Client requests only the fields it needs. Reduces payload. GraphQL-like flexibility in REST.

GET /api/v1/products?fields=id,name,price
// Returns only: { id, name, price } — not the full object
REST best practices
Use plural nouns (/products not /product) Keep URLs hierarchical and consistent Version your APIs (/v1, /v2) Support filtering, sorting, pagination Return appropriate status codes Never use verbs in URLs (GET /users/:id not /getUser) Consistent response envelope Use HTTPS always Never return unbounded lists Never put credentials in URLs Never return 200 for errors
API design
HTTP methods

Each HTTP method maps to a CRUD operation. Safe means no server state change. Idempotent means repeating has the same effect as doing it once.

GET
Read — retrieve a resource or collection
Safe and idempotent. Never change state with GET. GET /api/v1/products — get all. GET /api/v1/products/123 — get one. Query params for filtering.
POST
Create — make a new resource
Not safe, not idempotent — calling it twice creates two resources. POST /api/v1/products with body. Returns 201 Created + Location header pointing to new resource.
PUT
Update (replace) — full replacement
Idempotent — same result if repeated. Replaces the entire resource. PUT /api/v1/products/123 with full body. Missing fields become null/default.
PATCH
Update (partial) — modify specific fields
Not safe, not idempotent. Send only the fields to change — others remain. PATCH /api/v1/products/123 with partial body. Preferred over PUT for most updates.
DELETE
Delete — remove a resource
Idempotent — deleting something that doesn't exist is still OK. DELETE /api/v1/products/123. Returns 204 No Content. Consider soft-delete (mark as deleted) over hard-delete.
HEAD
Like GET — but no body in response
Used to check if resource exists, get metadata, or validate cache freshness. Same headers as GET. Used by crawlers and health checks.
OPTIONS
Describe available methods
Returns allowed methods for this endpoint. Used by browsers in CORS preflight requests before sending cross-origin POST/PUT/DELETE.
Safe vs idempotent

Safe methods

Does not change server state. GET, HEAD, OPTIONS are safe — you can call them 1000 times, nothing changes. Safe implies idempotent. Use for read operations only.

Idempotent methods

Calling multiple times has the same effect as calling once. GET, PUT, DELETE, HEAD are idempotent. POST is NOT — calling POST twice creates two resources. Critical for retry logic in distributed systems.

Good error response shape
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid product ID",
    "field": "productId",
    "details": [...],
    "request_id": "req_abc123"
  }
}
Rule: Always return a machine-readable error code + a human-readable message. Include a request_id for debuggability. Never return raw stack traces in production.
Critical distinctions

400 vs 422: 400 = malformed JSON or missing required field. 422 = valid JSON but business logic validation failed (email format wrong, age out of range).

401 vs 403: 401 = "I don't know who you are." Not authenticated. Fix: send your token. 403 = "I know who you are, but you can't do this." Fix: request higher permissions.

404 vs 403 (security): Reveal 403 = confirm resource exists. To hide resource existence from unauthorized users, return 404 instead of 403.

Never return 200 for errors. Some APIs (GraphQL, legacy RPC) return 200 with an error payload — this breaks HTTP caching, monitoring, and client error handling. Don't do this in REST.
API design
GraphQL APIs

Created by Facebook in 2012, open-sourced 2015. Solves REST's over-fetching and N+1 request problems with a typed query language and a single endpoint.

GraphQL is a query language for APIs. The client asks for exactly what it needs — nothing more. A single endpoint handles all operations. The schema is the contract between client and server — self-documenting and introspectable.
The problem GraphQL solves

REST: 4 requests for one screen

GET /api/users/123
GET /api/users/123/posts
GET /api/users/123/comments
GET /api/users/123/likes
→ 4 round trips, wasted data, N+1 problem

GraphQL: 1 request, exact data

query {
  user(id: "123") {
    name
    posts { title }
    comments { text }
  }
} → 1 round trip, only what you asked
Schema — the contract
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Query {
  user(id: ID!): User
  users(limit: Int = 20): [User!]!
}

type Mutation {
  createUser(name: String!, email: String!): User
  deleteUser(id: ID!): Boolean
}

type Subscription {
  userUpdated(id: ID!): User
}
Operations

Query (read)

Fetch data. Client specifies exact fields. Equivalent to GET in REST. Can fetch nested relations in one shot. Supports fragments for reusable field sets.

Mutation (write)

Create, update, or delete data. Equivalent to POST/PUT/DELETE in REST. Can return the updated object. Supports input types for complex arguments.

Subscription (real-time)

WebSocket-based. Server pushes updates when data changes. Live feeds, notifications, collaborative features. Not available in all GraphQL servers.

Error handling in GraphQL
GraphQL always returns 200 OK — even for errors. You must check the errors field in the response body. This is a common gotcha coming from REST.
{
  "data": { "user": null },
  "errors": [{
    "message": "User not found",
    "path": ["user"],
    "extensions": { "code": "NOT_FOUND" }
  }]
}
GraphQL pitfalls

N+1 query problem

Resolving each user's posts individually fires N database queries. Fix: use DataLoader (batching resolver). Accumulates IDs, fires one bulk query.

Query depth attacks

Malicious query nests deeply: user → friends → friends → friends... causing exponential DB load. Fix: set max query depth, complexity limits, and query cost analysis.

Caching complexity

REST has built-in HTTP caching per URL. GraphQL all goes through POST /graphql — HTTP caching doesn't work. Need app-level caching (Apollo Client cache, persisted queries).

Over-permissive schemas

Exposing too many fields in schema gives attackers access to internal data. Apply field-level authorization — not just resolver-level. Check auth per field for sensitive data.

API design
API Versioning

Once clients depend on your API, you can never break them. Versioning lets you evolve the API while keeping old consumers working. It's a promise to your consumers.

API versioning is the practice of maintaining multiple versions of an API simultaneously so existing clients keep working while new clients can adopt improvements. The hardest part isn't the versioning strategy — it's knowing when a change is "breaking".
Breaking vs non-breaking changes

Breaking changes (require new version)

Removing a field Renaming a field Changing field type Removing an endpoint Changing URL structure Making optional required Changing auth mechanism Changing error codes

Non-breaking changes (safe to ship)

Adding a new field Adding a new endpoint Making required optional Adding new enum values Adding new optional params Performance improvements Adding new error codes
Versioning strategies
URL path
Request header
Query param
Content negotiation

URL path versioning (most common)

Version embedded in the URL path. Visible, explicit, easy to test with curl. Most widely used — Stripe, Twitter, GitHub all use this.

GET /api/v1/users/123
GET /api/v2/users/123
// v1 and v2 run simultaneously — consumers choose
Easy to understand. Works in browser URL bar. Logs show which version each client uses. Simple to route in reverse proxy (nginx, API gateway).
URLs should be stable. Embedding version makes the URL "impure" from a pure REST perspective. But pragmatically — this is the right choice for most APIs.

Custom request header

Client sends a custom header to request a specific version. URL stays stable. Used by GitHub's API for some versions.

GET /api/users/123
X-API-Version: 2
// or
Accept: application/vnd.yourapi.v2+json
Less visible. Harder to test with browser URL bar. Easy to forget the header and get unexpected version. Hard to share URLs that hit a specific version.

Query parameter versioning

Version passed as a query parameter. Easy to test but pollutes the URL. Often used as a fallback/override alongside URL versioning.

GET /api/users/123?version=2
GET /api/users/123?v=2
Not recommended for primary versioning strategy. Easy to accidentally omit. Caching systems may not key on query params by default.

Content negotiation (Accept header)

Client specifies the exact media type it accepts, including version. REST-purist approach. Used by GitHub's v3 API.

Accept: application/vnd.github.v3+json
Accept: application/vnd.api.v2+json
Most "correct" from REST theory but hardest to use. Requires clients to set headers. Hard to explore in browser. Most teams prefer URL path versioning.

API design
Pagination

Never return unbounded lists. Even if there are only 10 rows today, there will be 10 million tomorrow. Pagination is not optional — it's a hard requirement for production APIs.

Pagination limits the number of items returned per request and provides a mechanism to fetch the next "page" of results. Choice of strategy dramatically affects performance, correctness, and developer experience at scale.
Offset
Cursor (recommended)
Keyset
Page-based

Offset pagination

Skip N records, return next M. Simple to implement and understand. Works great for small datasets. UI can jump to any page.

GET /products?offset=40&limit=20
// SQL equivalent: SELECT * FROM products LIMIT 20 OFFSET 40

// Response:
{
  "data": [...],
  "pagination": {
    "total": 1250,
    "offset": 40,
    "limit": 20,
    "has_more": true
  }
}
Problems at scale: Large OFFSET scans all skipped rows. Page 10,000 requires DB to scan 200,000 rows. Also: if rows are inserted/deleted between requests, you get duplicate or skipped items.

Cursor pagination (Stripe, Slack, Twitter style)

Each response includes an opaque cursor pointing to the last item seen. Next request starts from there. O(1) regardless of depth. Stable — inserts/deletes don't affect the cursor position.

// First request:
GET /products?limit=20

// Response includes next_cursor:
{
  "data": [...],
  "pagination": {
    "next_cursor": "eyJpZCI6MTIzfQ==",
    "has_more": true
  }
}

// Next request:
GET /products?after=eyJpZCI6MTIzfQ==&limit=20
Best for: infinite scroll, API consumers that process data sequentially, high-volume datasets. Cursor is typically base64-encoded ID or timestamp. Can support both after and before for bidirectional navigation.

Keyset pagination

Similar to cursor but uses actual column values instead of opaque tokens. Uses WHERE id > last_id SQL pattern. Performant — uses index scan not full table scan.

GET /products?after_id=123&limit=20
// SQL: WHERE id > 123 ORDER BY id ASC LIMIT 20

// Can also use timestamp:
GET /events?after=2024-01-01T00:00:00Z&limit=50
Transparent to consumers (no opaque cursor). Works well for time-series data. Requires a stable sort column with an index. Doesn't handle multi-column sort as cleanly as cursor pagination.

Page-based pagination

User-friendly page numbers. Simple for UI components with page numbers. Backed by offset under the hood.

GET /products?page=3&per_page=20

{
  "data": [...],
  "meta": {
    "current_page": 3,
    "total_pages": 63,
    "total_count": 1250,
    "per_page": 20
  }
}
Same offset-at-scale problems. Total count query can be expensive on large tables (COUNT(*) scans table). Avoid for APIs with millions of records.

The fastest API call is the one you don't make. Caching dramatically reduces latency, server load, and costs. HTTP has built-in caching primitives — use them.

Cache-Control directives

The primary HTTP caching header. Controls who can cache, for how long, and under what conditions.

// Public — CDN + browser can cache
Cache-Control: public, max-age=3600

// Private — only browser caches (user-specific data)
Cache-Control: private, max-age=300

// Must revalidate before serving stale
Cache-Control: public, max-age=600, must-revalidate

// Never cache (auth endpoints, mutations)
Cache-Control: no-store

// Cache but always revalidate with origin
Cache-Control: no-cache

// stale-while-revalidate: serve stale, refresh in background
Cache-Control: public, max-age=60, stale-while-revalidate=600

ETag — conditional GET

Server provides an opaque token (ETag) representing the resource state. Client sends it back — server responds 304 Not Modified if unchanged. Saves bandwidth without time-based expiry.

// Server response:
HTTP/1.1 200 OK
ETag: "v2-abc123"
Cache-Control: max-age=60

// Client's next request:
GET /products/123
If-None-Match: "v2-abc123"

// Server: unchanged → 304 Not Modified (no body)
// Server: changed → 200 OK + new ETag + new body

Last-Modified — time-based revalidation

Server sends when the resource was last modified. Client sends back in If-Modified-Since. Works for time-tracked resources.

// Server response:
Last-Modified: Mon, 01 Jan 2024 12:00:00 GMT

// Client's conditional request:
If-Modified-Since: Mon, 01 Jan 2024 12:00:00 GMT

// 304 if unchanged, 200 + new data if changed

Vary header — cache key dimensions

Tells caches which request headers affect the response. Critical for APIs returning different content based on headers (Accept-Language, Authorization, Accept-Encoding).

// Different languages = different cache entries
Vary: Accept-Language

// Different encodings
Vary: Accept-Encoding

// NEVER do this — effectively disables CDN caching
Vary: Authorization
// Each user would get their own CDN cache entry

Rate Limiting

Rate limiting protects your API from abuse, DDoS, and accidental over-use. Every production API must have it. The algorithm choice determines fairness and burst behavior.

Rate limiting restricts how many requests a user, IP, or API key can make in a given time window. Exceeded limits return 429 Too Many Requests. Headers communicate remaining quota to consumers.
Rate limit response headers
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1704067200  // Unix epoch of reset time
Retry-After: 60  // seconds until retry is safe

{ "error": { "code": "RATE_LIMIT_EXCEEDED", "message": "Retry in 60s" } }
Rate limiting algorithms
Token bucket
Leaky bucket
Fixed window
Sliding window

Token bucket

A bucket holds tokens. Each request consumes one token. Tokens refill at a fixed rate. If bucket is empty → reject. Allows bursts up to bucket size. Used by AWS, Stripe.

// Config: bucket capacity=100, refill=10 tokens/sec
request_1 → tokens: 99 (allowed)
request_2 → tokens: 98 (allowed)
...
request_100 → tokens: 0 (allowed)
request_101 → REJECTED (429)
// After 1 sec: tokens refill by 10 → 10 available
Best for: APIs that need to allow short bursts (mobile apps reconnecting, batch operations). Most flexible algorithm. Easy to tune per-user or per-endpoint.

Leaky bucket

Requests enter a queue (bucket). Processed at a fixed output rate. Queue full → reject. Smooths out traffic — no bursts allowed. Like a bucket with a hole leaking at constant rate.

// Config: leak rate = 10 req/sec, queue capacity = 100
incoming burst of 50 req → queued
output: processed at exactly 10/sec
// No burst allowed on output — constant smooth rate
Queue full (100 items) → REJECTED (429)
Best for: Protecting downstream services that can't handle bursts (DBs, payment processors). Produces consistent load on downstream.

Fixed window

Count requests in fixed time windows (e.g., per minute). Reset counter at window boundary. Simple but has boundary vulnerability.

// 100 req/min, window resets every minute
12:00:00–12:00:59 → 100 requests used
13:00:00–13:00:01 → 100 more requests → 200 req in 2 sec!
Vulnerability: Client can send 100 requests at 12:00:59 and 100 more at 13:00:01 — 200 requests at the boundary. Not suitable for strict rate limiting.

Sliding window

Track request timestamps in a rolling window. At any moment, count requests in the last N seconds/minutes. No boundary vulnerability. More accurate but more memory/CPU.

// 100 req/min sliding window
Now: 12:01:30
Window: 12:00:30 → 12:01:30 (last 60 sec)
Requests in window: 45
Remaining: 55
// Fair and accurate — no boundary attacks possible
Best general-purpose algorithm. Used by Redis ZSET with timestamp scoring. Slightly more expensive than fixed window but eliminates boundary attacks.
Rate limiting strategies

Per user

Limits per authenticated user ID. Fair — power users can't starve others. Most common for user-facing APIs.

Per API key

Different limits per tier. Free: 100/min. Pro: 1000/min. Enterprise: custom. Enables monetization of API access.

Per IP

Limit by source IP. Good for unauthenticated endpoints. Problem: shared IPs (NAT, offices) — limit hits entire company. Use as secondary signal, not primary.

Per endpoint

Different limits per route. /login: 5/min (brute-force protection). /search: 30/min. /export: 2/hour. Fine-grained control per resource sensitivity.

Per tenant

In multi-tenant SaaS, limit per organization not per user. All users in org share the org's quota. Enterprise plans get higher limits.

Global

Total requests across all users capped. Protects against systemic overload. Circuit breaker at the system level. Last resort protection.

What an API Gateway does

Authentication & Authorization

Validates JWT tokens, API keys, or OAuth tokens before the request reaches any service. Services trust that the gateway already verified identity. Centralizes auth logic.

Rate limiting

Per-key, per-IP, per-endpoint rate limiting at the edge. Drops excess requests before they hit services. Shared Redis state across gateway instances.

Handles HTTPS at the gateway. Backend services can communicate over plain HTTP within the private network. Manages TLS certificates in one place.

Request routing

Routes /api/users/* to user-service, /api/orders/* to order-service. Can do A/B routing, canary deployments, and blue-green releases at the gateway level.

Request/response transformation

Transform requests before forwarding (add headers, translate protocols). Transform responses before returning (aggregate multiple service responses, strip internal fields).

Observability

Centralized logging, request tracing, metrics collection. Every request logged once at the gateway. Correlation IDs added for distributed tracing across services.

Webhooks

Webhooks invert the polling model. Instead of your client asking "anything new?", the server tells your client "here's what just happened." Event-driven and efficient.

Webhooks are user-defined HTTP callbacks. When an event occurs (payment completed, file uploaded, user registered), the server sends a POST request to a client-registered URL. The client's server receives and processes it.
Polling vs Webhooks

Polling (inefficient)

Client asks every 30 seconds: "Did the payment complete?" 99% of responses: "No." Wastes API quota, adds latency, and scales poorly.

Webhook (event-driven)

Client registers URL. Server fires POST when payment completes. Instant notification. No wasted requests. Scales to millions of events per day.

Webhook payload (Stripe style)
// Server POSTs to your registered endpoint:
POST https://yourapp.com/webhooks/stripe
Content-Type: application/json
Stripe-Signature: sha256=abc123...

{
  "id": "evt_abc123",
  "type": "payment_intent.succeeded",
  "created": 1704067200,
  "data": { "object": { ... } }
}
Webhook security — critical

Verify the signature

Any attacker can POST to your webhook endpoint. Always verify the HMAC-SHA256 signature to confirm the request came from the legitimate sender. Never skip this step.

// Node.js verification:
const hmac = crypto.createHmac('sha256', secret);
hmac.update(rawBody);
const computed = hmac.digest('hex');
if (computed !== receivedSignature) {
  return res.status(401).send('Invalid signature');
}

Handle idempotency

Webhooks can be delivered more than once (network timeouts, retries). Your handler must be idempotent — processing the same event twice should not cause duplicate side-effects (double charges, double emails).

// Track processed event IDs:
if (await db.events.exists(event.id)) {
  return res.status(200).send('Already processed');
}
await processEvent(event);
await db.events.markProcessed(event.id);
Webhook best practices
Return 200 fast — process async via queue Verify HMAC signature on every request Make handlers idempotent Log all webhook events Implement retry logic with exponential backoff Support webhook secret rotation Never trust the payload without verification Don't do heavy processing in the handler
Delivery tip: Return 200 immediately, enqueue the event, process async. If your webhook handler takes >5 seconds, the sender will timeout and retry — causing duplicates.

OpenAPI / Swagger

OpenAPI is the standard machine-readable contract for REST APIs. Write once — generate docs, SDKs, mock servers, and test suites automatically.

OpenAPI Specification (OAS) is a standard, language-agnostic interface to REST APIs. Written in JSON or YAML. Describes endpoints, request/response schemas, auth, and examples. Swagger is the tooling ecosystem around it.
Minimal OpenAPI spec (YAML)
openapi: "3.1.0"
info:
  title: Product API
  version: "1.0.0"
  description: Manage products in the catalog

paths:
  /products/{id}:
    get:
      summary: Get a product by ID
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string }
      responses:
        "200":
          description: Product found
          content:
            application/json:
              schema: { $ref: "#/components/schemas/Product" }
        "404":
          description: Product not found

components:
  schemas:
    Product:
      type: object
      properties:
        id: { type: string }
        name: { type: string }
        price: { type: number }
What you get from an OpenAPI spec

Interactive docs

Swagger UI / ReDoc auto-generated. Consumers can try endpoints from the browser. No manual doc writing.

SDK generation

Generate type-safe client SDKs in any language from the spec. OpenAPI Generator supports 40+ languages. Consumers get a proper library, not raw HTTP calls.

Mock servers

Generate a mock server from the spec for frontend teams before the API is built. Prism, WireMock, Stoplight Studio.

Contract testing

Validate that your implementation matches the spec on every commit. Dredd, Schemathesis, Spectral.

Linting

Spectral enforces API design standards (no 200 for errors, consistent naming, no missing pagination). Run in CI.

API portal

Host a developer portal (Readme.io, Stoplight) from your OpenAPI spec. Track API usage, manage API keys, provide getting-started guides.

Design-first workflow: Write OpenAPI spec → review with team → generate mock server → build frontend against mock → build backend → run contract tests → ship.

Authentication (AuthN)

Authentication answers: "Who are you?" It's the first gate before authorization. Without it, you can't know what a user is allowed to do.

Authentication = verifying identity. Who is this person or system? Happens before authorization. Login request → system confirms identity → then authorization decides what they can do.
Basic auth
Bearer token
OAuth2 + JWT
SSO
API keys
mTLS

Basic authentication

Username + password sent as Base64-encoded string in the Authorization header on every request. Base64 is NOT encryption — it's trivially reversible. Requires HTTPS to be safe at all.

Authorization: Basic dXNlcjpwYXNzd29yZA==
// = base64("user:password") — easily decoded
Rarely used today except for internal tools and legacy systems. Credentials sent with every request. No token expiry — compromise lasts forever. Only use with HTTPS, never over HTTP.

Bearer token

Client receives a token at login. Sends it with every subsequent request in the Authorization header. Server validates the token. Stateless — server doesn't need to look up sessions in a database.

Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
// Standard approach in modern API design. Fast and stateless.
Standard approach today. Tokens can expire (unlike passwords). Can be revoked. Work across microservices without shared session state. Always store in httpOnly cookies or secure storage — never localStorage for sensitive tokens.

OAuth2 + JWT

OAuth2 is a delegation protocol — grants apps access to resources on your behalf without sharing your password. JWT (JSON Web Token) is the token format — self-contained, signed, stateless. Header.Payload.Signature.

JWT payload (decoded):
{
  "user_id": "123",
  "roles": ["admin", "editor"],
  "exp": 1720000000,  // expiry timestamp
  "iss": "auth.example.com",
  "jti": "unique-token-id"  // for revocation
}

Access token

Short-lived (15–60 min). Used for API calls. If stolen, expires quickly. Stored in memory or httpOnly cookie. Never in localStorage.

Refresh token

Long-lived (days/weeks). Gets new access tokens. Stored server-side or in httpOnly cookie. User doesn't have to re-login. Rotate on each use.

JWT caveat: JWTs can't be invalidated before expiry without a blocklist (because they're stateless). Keep access tokens short-lived. Maintain a token blocklist for logout/revocation.

SSO — Single Sign-On

Login once, access many apps. Google login → Gmail, Drive, Calendar all work. Enterprise users login to AD once → all internal tools work. Reduces password fatigue and attack surface.

SAML 2.0

Enterprise SSO. XML-based. Widely used with Salesforce, Okta, Azure AD. Older but still dominant in enterprise. Identity provider (IdP) issues assertions to service providers (SP).

OpenID Connect (OIDC)

Modern SSO. "Login with Google/GitHub." JSON-based. Built on top of OAuth2. OIDC adds identity layer — provides ID token (who you are) alongside access token (what you can do).

API keys

Long-lived secret tokens issued to developers/integrations. Passed in header or query param. No expiry by default. Used for server-to-server communication where OAuth is overkill.

X-API-Key: sk_live_abc123xyz...
// or
Authorization: Bearer sk_live_abc123xyz...
Good for: Server-to-server. CI/CD pipelines. Long-running integrations. Easier than OAuth2 for simple cases.
Risks: Harder to rotate. No per-request context. If leaked — full access until revoked. Scope them (read-only, write-only). Support rotation without downtime.

mTLS — Mutual TLS

Both client AND server authenticate with certificates. Server verifies client cert, client verifies server cert. Zero-trust architecture for internal microservices. No passwords, no tokens — cryptographic identity only.

// Regular TLS: only server has cert (client trusts server)
// mTLS: both have certs (mutual authentication)

Client → presents client cert
Server → validates against trusted CA
Server → presents server cert
Client → validates against trusted CA
→ Encrypted, mutually authenticated connection
Use mTLS for: Internal microservice-to-microservice calls. Zero-trust networks (no "inside" is trusted). Service mesh (Istio, Linkerd auto-injects mTLS). Compliance requirements (PCI-DSS, HIPAA).

Authorization (AuthZ)

Authorization answers: "What can you do?" It comes after authentication. The system knows who you are — now it decides your permissions.

Authorization = what actions is this authenticated user allowed to perform? Controls security, privacy, and system integrity. Three main models — each with different trade-offs.
RBAC
ABAC
ACL
OAuth2 scopes
RBAC — Role-Based Access Control. Assign users to roles. Roles have permissions. Simple and scalable for most use cases. The most widely used model.

Admin

Full access. Read, write, delete, manage users, configure system. Treat admin roles with principle of least surprise — require re-auth for destructive actions.

Editor

Create and edit content. Cannot delete. Cannot manage users. Typical for content contributors in a CMS.

Viewer

Read-only access. Cannot modify anything. Good default for new users or external collaborators.

GitHub teamsCMS tools (Contentful, Sanity)Slack workspacesGoogle Workspace
Principle of least privilege: Grant only the minimum permissions needed. Don't give Editor when Viewer suffices. Audit and remove unused permissions regularly.
ABAC — Attribute-Based Access Control. Decisions based on attributes of the user, the resource, and the environment. More flexible than RBAC. More complex to manage.
Example policy — allow access if:
user.department == "HR"
&& time < 18:00
&& resource.confidentiality == "internal"
&& user.clearance_level >= resource.required_clearance
Strengths: Extremely fine-grained control. Can express time-based, location-based, and context-aware policies. Scales to complex real-world rules.
Challenges: Policy conflicts are hard to detect. Testing all combinations is hard. Requires tooling (OPA, Casbin) to manage and audit policies safely.
Healthcare systemsFinancial servicesGovernment systemsAWS IAM policies
ACL — Access Control List. Each resource has its own explicit list of who can access it and how. Maximum granularity. User-facing permission sharing models.

Per-file permissions (Google Drive / Notion model)

AliceOwner — Read + Write + Share + Delete
BobEditor — Read + Write
CarolViewer — Read only
DaveNo access — cannot see resource exists
Granular control, but very hard to scale. Checking permissions for millions of files is expensive. Need indexing strategies (reverse ACL lookup). Good for user-facing sharing features.
OAuth2 delegated authorization with scopes. A third-party app requests access to your resources on another service. You approve specific scopes — not your full account. Scope = named permission.
GitHub OAuth — app requests scopes:
scope: "read:repo"   // can read repos
scope: "write:repo"  // can push code
scope: "read:user"   // can read profile
// NOT granted: delete:repo, admin:org
Key insight: The access token represents approved permissions, not a password. GitHub never shares your password. If the app is compromised, you revoke the token — not your whole account. Scopes enable the principle of least privilege for third-party access.

Use resource:action format (users:read, orders:write)Fine-grained scopes over broad onesDocument all scopes in your API referenceAvoid scope creep — don't ask for more than needed

Security techniques

Security is not a feature — it's a foundation. Every API exposed to the internet needs all of these. Add them at the start, not as an afterthought.

1. Rate limiting

Limit how many requests a user/IP can make in a given time window. Per endpoint, per user, per IP. Prevents DDoS attacks, API abuse, and brute-force attacks. Return 429 with Retry-After header.

Per-endpoint limits:
/login: 5 req/min   // brute-force protection
/search: 60 req/min
/export: 2 req/hour // expensive operation
→ Return 429 + Retry-After: 60

2. CORS — Cross-Origin Resource Sharing

Browsers block requests from one domain to another by default (same-origin policy). CORS headers tell browsers which domains are allowed to call your API. Prevents malicious websites from making API calls on behalf of logged-in users (CSRF via CORS).

Access-Control-Allow-Origin: https://app.yourdomain.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Authorization, Content-Type
// Never use "*" in production for authenticated APIs

3. SQL / NoSQL injection

Attacker sends malicious input that gets executed as a database query. Classic: entering admin'-- as a username to bypass password checks. Still #1 on OWASP Top 10.

Vulnerable:
"SELECT * FROM users WHERE name='" + input + "'"
// Input: admin'-- → bypasses password

Safe — parameterized query:
db.query("SELECT * FROM users WHERE name = ?", [input])

Safe — ORM:
User.findOne({ where: { name: input } })
Fix: Always use parameterized queries or ORM. Never concatenate user input into queries. Use allowlists for any dynamic column/table names.

4. Firewalls / WAF

Web Application Firewall sits in front of your API and filters malicious traffic before it reaches your servers. Blocks known attack patterns (OWASP Top 10), suspicious IPs, and bot traffic. Works at layer 7 (application layer) — understands HTTP.

AWS WAFCloudflare WAFModSecurityAzure Front Door

5. Input validation & sanitization

Validate every piece of input before using it. Type check, length limits, format validation, allowlists over blocklists. Sanitize before storing or rendering. Defense-in-depth alongside parameterized queries.

Validate:
email: must match RFC 5322 pattern
age: must be integer between 0 and 150
name: max 100 chars, strip HTML
file: allowed types: [jpg, png, pdf], max 10MB
// Reject early — return 400 with field-level errors

6. CSRF — Cross-Site Request Forgery

Attacker tricks a logged-in user's browser into making an unintended API call (e.g., "transfer $1000") using their existing session cookie. Browser automatically sends cookies — attacker's site exploits this.

Fix 1 — CSRF token (synchronizer token pattern):
X-CSRF-Token: <random_server_issued_token>

Fix 2 — SameSite cookie attribute:
Set-Cookie: session=abc; SameSite=Strict; Secure; HttpOnly

Fix 3 — Custom request header (CORS blocks cross-origin headers)
X-Requested-With: XMLHttpRequest

7. XSS — Cross-Site Scripting

Attacker injects malicious JavaScript into your app (e.g., in a comment field) that runs in other users' browsers — stealing cookies, tokens, or redirecting users. Stored XSS (persisted in DB) is the most dangerous.

Fix: Always escape user input before rendering. Use Content Security Policy (CSP) headers. Never use innerHTML with user-provided data. Use textContent instead. Set HttpOnly on auth cookies so JS can't steal them.
CSP header to restrict script sources:
Content-Security-Policy: default-src 'self';
  script-src 'self' cdn.example.com;
  object-src 'none'

8. HTTPS everywhere + HSTS

Always use HTTPS. HTTP Strict Transport Security (HSTS) forces browsers to only connect via HTTPS — prevents downgrade attacks where attacker degrades connection to HTTP.

HSTS header:
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
// 1 year — browser will never use HTTP again for this domain

9. VPN access controls

For internal APIs, require requests to come from within a VPN. Limits API access to known, trusted networks. Especially important for admin APIs and internal microservices that should never be publicly accessible.
