API Design — complete reference

APIs are the contracts between software components. This guide covers every layer — from transport protocols up to security patterns, versioning, and real-world design decisions.

What is an API?

Application Programming Interface. A documented contract that defines what requests can be made, how to make them, and what responses to expect. The interface between client and server.

API as contract

The API contract defines: valid requests, required parameters, authentication method, response shape, error codes. Break the contract = you break every consumer simultaneously.

Why API design matters

A bad API is forever. Once developers integrate your API, changing it breaks them. Good design reduces support burden, improves DX, enables scaling, and prevents security holes.

API consumers

Internal teams (microservices), external developers (third-party integrations), mobile apps, browsers, IoT devices, partner systems. Each has different needs and constraints.

API design quality spectrum

Bad API

Inconsistent naming. No versioning. 200 for all errors. No pagination. Returns entire object graph. Undocumented. No rate limiting. Changing behavior without notice.

OK API

Basic REST conventions. Some error codes correct. Some docs. No guarantees on stability. Versioning exists but inconsistent. Limited pagination support.

Great API

Consistent naming. Full OpenAPI docs. Proper status codes. Versioned with deprecation policy. Cursor pagination. Rate limit headers. Idempotency keys. Predictable evolution.

Design
4 key principles of great APIs

Every lasting API decision traces back to one of these four. Violate any one and you create technical debt that compounds over time.

1. Consistency

Same naming patterns everywhere. Same error shape everywhere. Same auth mechanism everywhere. Developers build mental models from patterns; break the pattern and you break their trust.

Consistent namingConsistent error shapesConsistent auth

2. Simplicity

Focus on core use cases. Every extra parameter, every optional field, every implicit behavior is surface area that must be maintained forever. An API is easy to use when it's hard to misuse.

One resource per endpointMinimal required paramsPredictable behavior

3. Security

Authentication — who are you? Authorization — what can you do? Input validation — is this data safe? Rate limiting — are you abusing the system? Security must be designed in from the start.

AuthN + AuthZ firstValidate all inputRate limit everything

4. Performance

Caching — serve from cache when fresh. Pagination — never return unbounded lists. Minimize payload size. Reduce round trips. Compress responses. One slow endpoint kills perceived performance.

Mandatory paginationHTTP cachingMinimize round trips

Protocols
HTTP / HTTPS

The foundation of data exchange on the web. Every REST API runs on HTTP. Understanding request/response structure is non-negotiable.

HTTP — a stateless, text-based application protocol. Client sends a request with method + URL + headers + body. Server responds with status code + headers + body. No memory of previous requests.
Request / response anatomy

HTTP request

GET /api/products/123 HTTP/1.1
Host: api.example.com
Authorization: Bearer tk9382nf...
Accept: application/json
Accept-Encoding: gzip

HTTP response

HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=3600
ETag: "abc123"
X-RateLimit-Remaining: 99

{ "id": 123, "price": 19.99 }
HTTPS = HTTP + TLS/SSL
Benefits: data encrypted in transit, data integrity, server authentication, SEO benefits. Required for any authentication to be secure.
Risks without: man-in-the-middle attacks, data tampering, credential exposure, session hijacking. Never ship a production API over plain HTTP.
Key headers

Content-Type

Format of the body. application/json, multipart/form-data. Must be set on requests with body and on all responses.

Authorization

Bearer <token> for JWT/OAuth, Basic <base64>. Never put credentials in URL query params.

Cache-Control

max-age=3600 = cache 1hr. no-store = never cache. stale-while-revalidate = serve stale, refresh background.

ETag

Entity tag for conditional requests. Client sends If-None-Match → server returns 304 if unchanged. Saves bandwidth.

Rate limit headers

X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After. Always include so clients back off gracefully.

Idempotency-Key

Client sends UUID. Server deduplicates. Critical for payment APIs — prevents double charges on network retries.

Transport layer
TCP vs UDP

These transport protocols determine how data physically moves between machines. Everything above — HTTP, WebSockets, gRPC — runs on top of one or the other.

TCP — Transmission Control Protocol

Reliable but slower. Guarantees every packet arrives in order. Uses the 3-way handshake to establish connection.

3-way handshake
1. SYN client → server: "I want to connect"
2. SYN-ACK server → client: "OK, I'm ready"
3. ACK client → server: "Connection established!"
Banking APIsEmailHTTP/RESTWebSocketsgRPC

UDP — User Datagram Protocol

Fast but unreliable. Fire-and-forget. No handshake, no delivery guarantee, no ordering. Less overhead = lower latency.

Packet lost? UDP says "NO RESENDING." A frame skip is acceptable; a 300ms stutter to retransmit is not.
Video callsOnline gamingLive streamingDNSHTTP/3 (QUIC)
Rule: If losing a packet hurts less than waiting for a retransmit — use UDP. If every byte must arrive — use TCP.

Protocols
WebSockets

Persistent, bidirectional connection. Replaces HTTP polling for real-time applications. One connection, unlimited messages in both directions.

WebSocket — upgrades an HTTP connection into a persistent, full-duplex channel. Both sides push data at any time. Runs over TCP.

HTTP polling (old way)

Client asks "Any new data?" → "No." → asks again → "No." Wasted bandwidth, added latency, not truly real-time.

Always 1+ polling interval behind. Wastes bandwidth on empty responses.

WebSocket (right way)

One HTTP handshake → upgrades to WebSocket → stays open forever. Server pushes the instant data is available.

Real-time, minimal latency. Bidirectional over single connection.
WebSockets vs SSE

WebSockets (bidirectional)

Full-duplex — both sides send. Better for chat, gaming, collaborative editing. Needs custom reconnection logic.

SSE — Server-Sent Events (server→client only)

One-way server push. Automatic reconnection built-in. Works through proxies. Good for dashboards, live feeds, notifications.

Use cases
Chat apps (WhatsApp, Slack)Live tradingMultiplayer gamesLive collaboration (Figma)Real-time notificationsLive sports

Protocols
gRPC — Google Remote Procedure Call

High-performance, binary protocol for server-to-server communication. Used heavily in microservices. Up to 10x faster than JSON/REST.

gRPC uses Protocol Buffers (protobuf) for serialization and HTTP/2 as transport. Services defined in .proto files. Client stubs auto-generated in any language.

Protocol Buffers (.proto)

Binary format — ~5–10x smaller than JSON, faster to parse. Type-safe — schema mismatch caught at compile time.

// user.proto
syntax = "proto3";

message User {
  string id = 1;
  string name = 2;
  int32 age = 3;
}

service UserService {
  rpc GetUser (UserRequest) returns (User);
}

4 communication types

Unary — 1 req, 1 res (like REST).
Server streaming — 1 req, many res.
Client streaming — many req, 1 res.
Bidirectional — both sides stream simultaneously.

Protocols
Choosing the right protocol

Choosing the wrong protocol creates a bottleneck. Most systems use multiple protocols for different needs.

Interaction pattern

Request-response → HTTP/REST
Real-time push → WebSockets
Async fire-and-forget → AMQP / Kafka
Service-to-service streaming → gRPC
Server → client only → SSE

Performance

gRPC (binary, HTTP/2) is fastest for internal. REST (text, HTTP/1.1) has more overhead. WebSockets for sustained real-time. AMQP is async — latency tolerance required.

Client compatibility

Browsers → REST or GraphQL. Mobile → REST or gRPC. Internal services → gRPC. IoT → MQTT or AMQP. Third-party → REST (widest tool support).

Payload characteristics

Large binary → gRPC. Small JSON → REST. Complex nested data → GraphQL. Streaming large datasets → gRPC server streaming.

Security

Always TLS. gRPC has TLS via HTTP/2. HTTPS for REST. WSS for WebSockets. mTLS for zero-trust internal services.

Developer experience

REST has best tooling (Postman, Swagger, curl). GraphQL has Playground. gRPC requires proto toolchain. Consider team familiarity.

RESTful APIs

Resource-based URL design, filtering, sorting, pagination, and versioning — the complete REST playbook.

REST uses standard HTTP methods and resource-based URLs. 6 constraints: stateless, uniform interface, client-server separation, layered system, cacheable, code on demand (optional).

API design
HTTP methods

Each HTTP method maps to a CRUD operation. Safe = no state change. Idempotent = repeating has the same effect as doing it once.

GET
Read — retrieve a resource
Safe and idempotent. Never change state. GET /products = all. GET /products/123 = one.
POST
Create — make a new resource
Not safe, not idempotent — calling twice creates two resources. Returns 201 Created + Location header.
PUT
Update (replace) — full replacement
Idempotent. Replaces the entire resource. Missing fields become null/default.
PATCH
Update (partial) — modify fields
Send only the fields to change. Others remain. Preferred over PUT for most updates.
DELETE
Delete — remove a resource
Idempotent — deleting something that doesn't exist is still OK. Returns 204 No Content.
HEAD
Like GET — no body in response
Check if resource exists, get metadata, validate cache freshness.
OPTIONS
Describe available methods
Returns allowed methods. Used by browsers in CORS preflight requests.

Critical distinctions

401 vs 403: 401 = "I don't know who you are." 403 = "I know who you are, but you can't do this."

400 vs 422: 400 = malformed JSON or missing field. 422 = valid JSON but business logic validation failed.

Never return 200 for errors. This breaks HTTP caching, monitoring, and client error handling.
API design
GraphQL APIs

Created by Facebook 2012, open-sourced 2015. Solves REST's over-fetching and N+1 request problems with a typed query language.

GraphQL — client asks for exactly what it needs. Single endpoint handles all operations. Schema is the contract. Self-documenting via introspection.

REST: 4 requests for one screen

GET /api/users/123
GET /api/users/123/posts
GET /api/users/123/comments
GET /api/users/123/likes
→ 4 round trips, wasted data

GraphQL: 1 request, exact data

query {
  user(id: "123") {
    name
    posts { title }
    comments { text }
  }
} → 1 round trip
Operations

Query (read)

Fetch data. Client specifies exact fields. Equivalent to GET.

Mutation (write)

Create, update, or delete data. Equivalent to POST/PUT/DELETE.

Subscription (real-time)

WebSocket-based. Server pushes updates when data changes.

GraphQL always returns 200 OK — even for errors. You must check the errors field. Common gotcha from REST.
GraphQL pitfalls

N+1 query problem

Resolving each user's posts individually fires N database queries. Fix: DataLoader (batching resolver).

Query depth attacks

Deeply nested queries cause exponential DB load. Fix: max query depth, complexity limits.

Caching complexity

All goes through POST /graphql — HTTP caching doesn't work. Need app-level caching (Apollo Client, persisted queries).

Over-permissive schemas

Apply field-level authorization — not just resolver-level. Check auth per field for sensitive data.

API Versioning

Once clients depend on your API, you can never break them. Versioning lets you evolve the API while keeping old consumers working.

Breaking changes (require new version)

Removing a fieldRenaming a fieldChanging field typeRemoving an endpointChanging URL structureChanging auth mechanism

Non-breaking changes (safe)

Adding a new fieldAdding a new endpointMaking required optionalAdding new optional paramsPerformance improvements
Versioning strategies
URL path
Request header
Query param

URL path versioning (most common)

Version in the URL. Visible, easy to test with curl. Used by Stripe, Twitter, GitHub.

GET /api/v1/users/123
GET /api/v2/users/123
// v1 and v2 run simultaneously
Easy to understand. Logs show which version each client uses. Simple to route in API gateway.

Custom request header

Client sends a custom header. URL stays stable.

GET /api/users/123
X-API-Version: 2
Less visible. Harder to test. Easy to forget the header. Hard to share versioned URLs.

Query parameter versioning

Version as a query param. Easy to test but pollutes the URL.

GET /api/users/123?version=2
Not recommended for primary strategy. Easy to accidentally omit.

API design
Pagination

Never return unbounded lists. Even if there are only 10 rows today, there will be 10 million tomorrow. Pagination is non-optional.

Offset
Cursor (recommended)
Keyset

Offset pagination

Skip N records, return next M. Simple. Works for small datasets. UI can jump to any page.

GET /products?offset=40&limit=20
// SQL: SELECT * FROM products LIMIT 20 OFFSET 40

{ "data": [...], "pagination": { "total": 1250, "offset": 40, "limit": 20 } }
Problems at scale: Large OFFSET scans all skipped rows. Page 10,000 requires DB to scan 200,000 rows. Also: inserts/deletes cause duplicate or skipped items.

Cursor pagination (Stripe, Slack, Twitter style)

Each response includes an opaque cursor. Next request starts from there. O(1) regardless of depth. Stable under inserts/deletes.

GET /products?limit=20

// Response:
{ "data": [...], "pagination": { "next_cursor": "eyJpZCI6MTIzfQ==", "has_more": true } }

// Next request:
GET /products?after=eyJpZCI6MTIzfQ==&limit=20
Best for: Infinite scroll, feeds, high-volume datasets. Cursor is typically base64-encoded ID.

Keyset pagination

Uses actual column values. WHERE id > last_id. Performant — uses index scan not full table scan.

GET /products?after_id=123&limit=20
// SQL: WHERE id > 123 ORDER BY id ASC LIMIT 20
Transparent (no opaque cursor). Works well for time-series. Requires stable sort column with index.

Security
Rate Limiting

Rate limiting protects your API from abuse, DDoS, and accidental over-use. Every production API must have it.

HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
Retry-After: 60

{ "error": { "code": "RATE_LIMIT_EXCEEDED", "message": "Retry in 60s" } }
Algorithms
Token bucket
Leaky bucket
Sliding window
Fixed window

Token bucket

Bucket holds tokens. Each request consumes one. Tokens refill at fixed rate. Allows bursts up to bucket size. Used by AWS, Stripe.

Best for: APIs that allow short bursts. Most flexible algorithm.

Leaky bucket

Requests enter a queue processed at fixed output rate. Smooths traffic — no bursts on output. Queue full → reject.

Best for: Protecting downstream services that can't handle bursts.

Sliding window (recommended)

Track request timestamps in a rolling window. At any moment, count requests in the last N seconds. No boundary vulnerability. More accurate.

Best general-purpose algorithm. Implemented with Redis ZSET + timestamp scoring.

Fixed window

Count requests in fixed time windows. Reset counter at window boundary. Simple but has boundary vulnerability.

Vulnerability: 100 requests at 12:00:59 + 100 more at 13:00:01 = 200 requests in 2 seconds. Not suitable for strict rate limiting.
Strategies

Per user

Limits per authenticated user ID. Fair — power users can't starve others.

Per API key / tier

Free: 100/min. Pro: 1000/min. Enterprise: custom. Enables monetization.

Per endpoint

/login: 5/min. /search: 60/min. /export: 2/hour. Fine-grained per resource sensitivity.

What an API Gateway does

Authentication & Authorization

Validates JWT tokens, API keys, OAuth before the request reaches any service. Services trust the gateway already verified identity.

SSL termination

Handles HTTPS at the gateway. Backend services communicate over plain HTTP within the private network. Manages TLS certificates in one place.

Request routing

Routes /api/users/* to user-service. A/B routing, canary deployments, blue-green releases at gateway level.

Observability

Centralized logging, request tracing, metrics. Every request logged once. Correlation IDs added for distributed tracing.

Don't: Put business logic in the gateway. Only infrastructure concerns. Logic there is hard to test, version, and debug.
API design
Webhooks

Instead of your client asking "anything new?", the server tells your client "here's what just happened." Event-driven and efficient.

Webhooks are user-defined HTTP callbacks. When an event occurs, the server sends a POST to a client-registered URL.

Polling (inefficient)

Client asks every 30 seconds: "Did the payment complete?" 99% responses: "No." Wastes quota, adds latency.

Webhook (event-driven)

Client registers URL. Server fires POST when payment completes. Instant. No wasted requests.

Webhook security — critical

Verify the signature

Any attacker can POST to your endpoint. Verify the HMAC-SHA256 signature on every request. Never skip this.

// Node.js:
const hmac = crypto.createHmac('sha256', secret);
hmac.update(rawBody);
const sig = hmac.digest('hex');
if (sig !== received) return res.status(401).send();

Handle idempotency

Webhooks can be delivered more than once. Processing the same event twice must not cause duplicates (double charges, double emails).

if (await db.events.exists(event.id)) {
  return res.status(200).send();
}
await processEvent(event);
Tip: Return 200 immediately. Enqueue the event. Process async. If handler takes >5s, sender times out and retries — causing duplicates.
API design
OpenAPI / Swagger

OpenAPI is the standard machine-readable contract for REST APIs. Write once — generate docs, SDKs, mock servers, and test suites automatically.

OpenAPI Specification (OAS) is a language-agnostic interface description for REST APIs in JSON or YAML. Swagger is the tooling ecosystem.
Minimal OpenAPI spec
openapi: "3.1.0"
info:
  title: Product API
  version: "1.0.0"

paths:
  /products/{id}:
    get:
      summary: Get a product by ID
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string }
      responses:
        "200": { description: Product found }
        "404": { description: Not found }
What you get from an OpenAPI spec

Interactive docs

Swagger UI / ReDoc auto-generated. Consumers try endpoints from browser.

SDK generation

Type-safe client SDKs in 40+ languages. OpenAPI Generator.

Mock servers

Frontend teams build against mocks before backend is built. Prism, Stoplight.

Contract testing

Validate implementation matches spec on every commit. Dredd, Schemathesis.

Linting

Spectral enforces design standards. Run in CI.

API portal

Host developer portal from spec. Readme.io, Stoplight.

Design-first workflow: Write OpenAPI spec → review → generate mock → build frontend against mock → build backend → run contract tests → ship.

Authentication (AuthN)

Authentication answers: "Who are you?" It's the first gate before authorization.

Authentication = verifying identity. Happens before authorization. Login → system confirms identity → authorization decides what they can do.
Basic auth
Bearer token
OAuth2 + JWT
SSO
API keys
mTLS

Basic authentication

Username + password Base64-encoded in Authorization header on every request. Base64 is NOT encryption. Requires HTTPS.

Authorization: Basic dXNlcjpwYXNzd29yZA==
// = base64("user:password")
Rarely used today. Credentials sent with every request. No token expiry.

Bearer token

Client receives token at login. Sends it with every request. Stateless — server doesn't look up sessions.

Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Standard approach today. Tokens expire. Can be revoked. Work across microservices without shared session state.

OAuth2 + JWT

OAuth2 grants apps access without sharing your password. JWT is self-contained, signed, stateless. Header.Payload.Signature.

JWT payload:
{ "user_id": "123", "roles": ["admin"], "exp": 1720000000 }

Access token

Short-lived (15–60 min). For API calls. If stolen, expires quickly.

Refresh token

Long-lived (days/weeks). Gets new access tokens. Rotate on each use.

JWT caveat: Can't be invalidated before expiry without a blocklist. Keep access tokens short-lived.

SSO — Single Sign-On

Login once, access many apps. Reduces password fatigue and attack surface.

SAML 2.0

Enterprise SSO. XML-based. Okta, Azure AD. Older but dominant in enterprise.

OpenID Connect (OIDC)

Modern SSO. "Login with Google/GitHub." JSON-based, built on OAuth2. Provides ID token (who) + access token (what).

API keys

Long-lived secret tokens. Server-to-server communication. Easier than OAuth2 for simple cases.

X-API-Key: sk_live_abc123xyz...
Good for: Server-to-server. CI/CD pipelines. Long-running integrations.
Risks: Harder to rotate. If leaked — full access until revoked. Scope them (read-only, write-only).

mTLS — Mutual TLS

Both client AND server authenticate with certificates. Zero-trust architecture. No passwords, no tokens — cryptographic identity only.

// Regular TLS: only server has cert
// mTLS: both have certs (mutual authentication)

Client → presents client cert → server validates
Server → presents server cert → client validates
→ Encrypted, mutually authenticated
Use for: Internal microservice calls. Zero-trust networks. Service mesh (Istio, Linkerd). Compliance (PCI-DSS, HIPAA).

Authorization (AuthZ)

Authorization answers: "What can you do?" It comes after authentication.

Authorization = what actions is this authenticated user allowed to perform? Three main models — each with different trade-offs.
RBAC
ABAC
ACL
OAuth2 scopes
RBAC — Role-Based Access Control. Assign users to roles. Roles have permissions. Simple and scalable.

Admin

Full access. Read, write, delete, manage users, configure system.

Editor

Create and edit content. Cannot delete. Cannot manage users.

Viewer

Read-only. Cannot modify anything.

GitHub teamsCMS toolsSlack workspaces
Principle of least privilege: Grant only minimum permissions needed. Audit and remove unused permissions regularly.
ABAC — Attribute-Based Access Control. Decisions based on attributes of user, resource, and environment. More flexible than RBAC. More complex.
Example policy:
user.department == "HR"
&& time < 18:00
&& resource.confidentiality == "internal"
HealthcareFinancial servicesGovernmentAWS IAM policies
ACL — Access Control List. Each resource has its own explicit list. Maximum granularity.

Per-file permissions (Google Drive model)

AliceOwner — Read + Write + Share + Delete
BobEditor — Read + Write
CarolNo access
Granular but hard to scale. Need indexing strategies. Good for user-facing sharing features.
OAuth2 scopes. Third-party app requests specific permissions. You approve scopes — not full account access.
GitHub OAuth scopes:
scope: "read:repo"   // read repos
scope: "write:repo"  // push code
// NOT granted: delete:repo
Key: Access token = approved permissions, not a password. If app is compromised, revoke the token — not your whole account.

Security techniques

Security is not a feature — it's a foundation. Every production API needs all of these.

1. Rate limiting

Limit requests per user/IP/key per time window. Prevents DDoS, brute-force, API abuse. Return 429 with Retry-After.

2. CORS

Tells browsers which domains can call your API. Prevents malicious sites from making API calls using user's session.

Access-Control-Allow-Origin: https://app.yourdomain.com
// Never "*" for authenticated APIs

3. SQL / NoSQL injection

Attacker sends malicious input executed as DB query. Still #1 on OWASP Top 10.

"SELECT * FROM users WHERE name='" + input + "'"
// VULNERABLE

db.query("SELECT * FROM users WHERE name=?", [input])
// SAFE — parameterized

4. Input validation

Validate every input. Type check, length limits, format validation, allowlists. Reject early with 400 + field-level errors.

5. CSRF

Attacker tricks logged-in user's browser into making unintended API call using session cookie.

X-CSRF-Token: <random_server_issued_token>
// Or: SameSite=Strict cookie attribute

6. XSS

Attacker injects malicious JS into your app. Runs in other users' browsers — steals cookies, tokens.

Fix: Escape user input before rendering. Use CSP headers. Never innerHTML with user data. HttpOnly cookies.

7. HTTPS + HSTS

Always HTTPS. HSTS forces browsers to only use HTTPS — prevents downgrade attacks.

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

8. WAF

Web Application Firewall filters malicious traffic at layer 7. Blocks OWASP Top 10 patterns, suspicious IPs, bot traffic.

AWS WAFCloudflare WAFAzure Front Door

9. VPN access controls

For internal APIs, require requests from within VPN. Limits access to known networks. Critical for admin APIs.

OWASP API Security Top 10
1
Broken Object Level Authorization
API returns data for IDs it shouldn't. Check ownership on every request.
2
Broken Authentication
Weak tokens, no expiry, credential stuffing.
3
Broken Object Property Level Auth
Exposing internal fields, mass assignment (set isAdmin=true).
4
Unrestricted Resource Consumption
No rate limiting, no payload size limits.
5
Broken Function Level Authorization
Admin endpoints accessible to normal users.
6
Unrestricted Sensitive Flows
No brute-force protection on login, OTP, password reset.
7
SSRF
API fetches URLs from user input — attacker targets internal services.
8
Security Misconfiguration
Debug endpoints exposed, verbose errors in prod.
9
Improper Inventory Management
Old API versions still live, undocumented endpoints.
10
Unsafe API Consumption
Blindly trusting third-party API data without validation.

API Design Checklist

Run through this before shipping any API to production. Most commonly missed requirements that cause pain in production.

URL & resource design

✓
Plural nouns for collections (/products not /product)
✓
No verbs in URLs
✓
Consistent naming (snake_case or camelCase — pick one)
✓
Versioned with /v1, /v2
✓
Nested resources max 2 levels deep

HTTP correctness

✓
Correct HTTP method for each operation
✓
Correct status codes (201 for create, 204 for delete)
✓
Location header on 201 Created responses
✓
Never 200 for errors

Security

✓
All endpoints require authentication (unless explicitly public)
✓
Authorization checked on every request
✓
Rate limiting on every endpoint
✓
HTTPS only — no HTTP fallback
✓
Input validation on every parameter
✓
CORS configured correctly (not *)

Data & responses

✓
Consistent response envelope across all endpoints
✓
All list endpoints have pagination
✓
Max page size enforced (cap at 100 or less)
✓
Machine-readable error codes in responses
✓
Request ID in all responses for debugging

Performance & caching

✓
Cache-Control headers on all GET responses
✓
ETag support for conditional GETs
✓
Gzip compression on responses > 1KB
✓
Rate limit headers on all responses

Documentation

✓
OpenAPI spec for all endpoints
✓
All error codes documented
✓
Request/response examples per endpoint
✓
Changelog for all breaking changes
✓
Deprecation notices with Sunset header
