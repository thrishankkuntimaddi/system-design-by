Chapter 1

Basic Introduction

Understanding the core architecture, types of operating systems, process transitions, and execution modes.

1. Introduction & Batch operating system
An **Operating System (OS)** is system software that acts as an intermediary between a user of a computer and the computer hardware. Its primary goals are to execute user programs, make the computer system convenient to use, and utilize hardware efficiently.

Batch Operating System

Definition: Users do not interact with the computer system directly. Instead, a user prepares their job (consisting of the program, data, and control cards) and submits it to a computer operator. The operator groups jobs with similar requirements into **batches** to speed up processing.
Pros: Reduces setup time for similar jobs; can run overnight without operator supervision.
Cons: Lacks direct interaction; CPU remains idle during slow I/O operations; debugging is difficult.

2. Types of Operating Systems

Multiprogramming

Keeps multiple jobs in memory simultaneously. When the current job halts for I/O, the CPU switches to another job. Prevents CPU idling.
Use Case: Early mainframe operating systems.

Multitasking (Time-Sharing)

An extension of multiprogramming where the CPU executes multiple jobs by switching among them rapidly (via time quantum clocks). Interactive user context.
Use Case: Modern OS (Windows, macOS, Linux).

Real-Time OS (RTOS)

Guarantees output within strict, predictable time boundaries.
• **Hard RTOS:** Missing a deadline results in system failure (e.g. missile guidance, pacemakers).
• **Soft RTOS:** Missing a deadline degrades quality but isn't fatal (e.g. video streaming).

Distributed & Clustered

• **Distributed:** Manages a collection of independent CPUs that appear to the user as a single system.
• **Clustered:** Combines multiple physical machines into a unified cluster to share storage and provide high availability.

3. Process States & Schedulers
A **Process** is a program in execution. Its lifecycle transitions through multiple states:

Long-Term Scheduler

Job scheduler. Selects processes from storage and loads them into memory. Controls the **degree of multiprogramming**.

Short-Term Scheduler

CPU scheduler. Selects a process from the Ready queue in memory and allocates the CPU to it. Must be extremely fast.

Medium-Term Scheduler

Swapping scheduler. Swaps processes out of memory to disk to reduce memory pressure, and swaps them back later.

4. Linux Commands & System Calls

User Mode vs. Kernel Mode

To protect hardware, execution is divided:
• **User Mode (Mode bit = 1):** Restricted access to memory and peripheral hardware.
• **Kernel Mode (Mode bit = 0):** Unrestricted access to hardware and instructions. Transitions to Kernel mode occur via **System Calls** (e.g. read, write, open).

The fork() System Call

Creates a new child process by duplicating the calling parent process.
• Returns `0` to the child process.
• Returns the child's process ID (PID) to the parent process.
• Returns a negative value if creation fails.

#include <stdio.h>
#include <unistd.h>

int main() {
    fork(); // Split: parent and child
    fork(); // Split again: now 4 processes
    printf("Hello\n"); // Prints 4 times
    return 0;
}
5. Processes vs. Threads
Characteristic Process Thread (Lightweight Process)
Definition Independent executing program instance. Segment of execution within a parent process.
Memory Sharing Separate virtual memory address spaces. Shares code, data, and file descriptors of parent.
Context Switch Slow, requires page table and cache flushes. Fast, only registers and stack are switched.
Isolation Strong isolation. One crash doesn't affect others. Weak isolation. One thread crash can crash process.

User-Level vs. Kernel-Level Threads

• **User-Level Threads:** Managed by user-space libraries. The OS is unaware of them. Fast to create and switch, but if one user thread blocks on I/O, the entire process blocks.
• **Kernel-Level Threads:** Managed directly by the OS. The kernel handles scheduling. If one thread blocks on I/O, the kernel can schedule another thread from the same process.

Process Scheduling

CPU allocation algorithms, Gantt chart scheduling evaluations, and queue strategies.

1. Scheduling Terms & Definitions
• Preemption: The OS can interrupt a running process to allocate CPU to another process.
• Non-preemption: A process holds the CPU until it terminates or blocks (voluntarily relinquishes control).
• Arrival Time (AT): Time when a process enters the ready queue.
• Burst Time (BT): CPU execution time required by the process.
• Completion Time (CT): Time when the process finishes execution.
• Turnaround Time (TAT): Total time from arrival to completion. TAT = CT - AT.
• Waiting Time (WT): Time spent waiting in the ready queue. WT = TAT - BT.
• Response Time (RT): Time from arrival to the first CPU allocation. RT = First Allocation - AT.

Definition: Non-preemptive scheduling where the process that requests the CPU first is allocated the CPU first, managed via a standard FIFO queue.

Definition: CPU scheduling where tasks are assigned priorities, and a higher priority process preempts a lower priority running process.

3. Mixed Burst & Multi-Level Queue

Mixed CPU & I/O Bursts

Processes alternate between CPU bursts and waiting on I/O devices (e.g. disk read, user input). Schedulers switch active processes when a process halts to wait on I/O, preventing the CPU from sitting idle.

Multi-Level Queue Scheduling

Partitions the ready queue into separate queues based on process type (e.g. interactive vs. batch). Each queue has its own scheduling algorithm.

Feedback Queue: Allows processes to move between queues based on their CPU usage. Prevents starvation for interactive jobs.

Chapter 3

Process Synchronization

Avoiding race conditions, identifying critical section criteria, locks, and semaphores.

1. Race Conditions & Critical Section
A **Race Condition** occurs when multiple processes access and manipulate the same shared data concurrently, and the outcome of the execution depends on the order in which the access takes place.

Printer-Spooler Problem

A directory contains spooler slots. Process A reads the next free slot index (say, 4). Before Process A can write its file, it is preempted. Process B runs, reads the free slot index (still 4), and writes its file. When Process A resumes, it overwrites Process B's file. This is a classic race condition.

Critical Section Requirements

The region of code accessing shared variables. A solution to the critical section problem must satisfy:

1. **Mutual Exclusion:** Only one process can execute in its critical section at a time.
2. **Progress:** If no process is executing in its critical section, only processes wishing to enter can participate in deciding which process enters next.
3. **Bounded Waiting:** There must be a limit on the number of times other processes can enter their critical sections before a process's request to enter is granted.

2. Synchronization Primitives

Lock Variables

A simple shared boolean variable. A process polls while(lock == 1), then sets lock = 1 before entering. Fails to guarantee mutual exclusion if preemption occurs immediately after the while loop evaluation.

Turn Variable (Strict Alternation)

Uses a shared variable turn to indicate which process's turn it is. Guarantees mutual exclusion, but fails progress because processes must wait even if the other process has no interest in entering.

TestAndSet Instruction

An atomic hardware instruction. It reads the lock value and sets it to true in a single, uninterruptible clock cycle, ensuring mutual exclusion without software race conditions.

3. Semaphores
A **Semaphore** is an integer variable accessed only through two atomic, uninterruptible operations:
• wait(S) or P(S): Decrements S. If $S \le 0$, the calling process blocks.
• signal(S) or V(S): Increments S. If processes are blocked, it wakes one up.

Binary Semaphore

Can only take values `0` or `1`. Acts exactly like a mutex lock. Used to guarantee mutual exclusion.

Counting Semaphore

Value can range over an unrestricted domain. Used to control access to a resource pool with a finite number of instances.

4. Classical Problems of Synchronization

Producer-Consumer Problem

Uses shared buffer slots. Uses three semaphores:
• mutex (binary, initialized to 1) protects buffer access.
• empty (counting, initialized to N) tracks empty slots.
• full (counting, initialized to 0) tracks filled slots.

Dining Philosophers Problem

5 philosophers sit around a table with 5 chopsticks. Each philosopher needs both adjacent chopsticks to eat. If all philosophers pick up their left chopstick simultaneously, a deadlock occurs (circular wait).

Solutions: Allow at most 4 philosophers to sit, or allow a philosopher to pick up chopsticks only if both are available.

Chapter 4

Deadlocks

Understanding deadlock conditions, prevention strategies, avoidance safe-states, and safety evaluations using the Banker's Algorithm.

1. Deadlock Conditions & Graphs
A **Deadlock** is a state where a set of processes are blocked because each process is holding a resource and waiting for another resource held by a different process in the set.

Four Coffman Conditions

All four must hold simultaneously for a deadlock to occur:

1. **Mutual Exclusion:** At least one resource must be held in a non-shareable mode.
2. **Hold and Wait:** A process must hold at least one resource and be waiting for another.
3. **No Preemption:** Resources cannot be preempted; they are released only voluntarily.
4. **Circular Wait:** Process P0 waits for a resource held by P1, which waits for P2, ..., which waits for P0.

Resource Allocation Graph (RAG)

A directed graph representing system state.
• Vertices: Processes (P) and Resources (R).
• Request Edge: Directed edge Pi → Rj (process is waiting for resource).
• Assignment Edge: Directed edge Rj → Pi (resource is allocated to process).

Rule: If the graph has no cycles, no deadlock exists. If a cycle exists: deadlock exists if resources have single instances; deadlock may exist if resources have multiple instances.

2. Strategies for Handling Deadlocks

1. Deadlock Prevention

Design the system to eliminate at least one of the four Coffman conditions:

• **Mutual Exclusion:** Make resources shareable (e.g. read-only files). Inherently impossible for spoolers/printers.
• **Hold and Wait:** Require processes to request all resources at start, or release held resources before making new requests. (Disadvantage: Low resource utilization).
• **No Preemption:** If a process requests a resource that is unavailable, it must release all its currently held resources.
• **Circular Wait:** Impose a global ordering on all resources. A process can only request resources in strictly increasing order (e.g. if holding R2, cannot request R1). Provably prevents cycles.

2. Deadlock Avoidance

OS dynamically decides whether to grant a resource request based on whether the resulting state is **Safe** or **Unsafe**.

• **Safe State:** There exists a safe execution sequence (e.g. P1, P2, ...) where all processes can eventually get their maximum required resources and terminate.
• **Unsafe State:** Not necessarily a deadlock, but the OS can no longer prevent processes from eventually deadlocking if they request their maximum resources.
• **Algorithm:** Banker's Algorithm is used for multi-instance resource types.

3. Deadlock Detection & Recovery

Allow deadlocks to occur, periodically detect them, and recover:

• **Detection:** OS builds a **Wait-For Graph** (collapsing RAG resources) and runs cycle-detection algorithms (e.g., DFS, finding cycles in O(V+E) time).
• **Recovery by Process Termination:** Abort all deadlocked processes (expensive), or abort one process at a time until the deadlock cycle is broken.
• **Recovery by Resource Preemption:** Temporarily seize resources from some processes. Requires: Victim Selection (minimize cost), Rollback (return to a safe checkpoint), and avoiding Starvation (ensure the same process isn't always victimized).

4. Deadlock Ignorance (Ostrich Algorithm)

Ignore the problem entirely. Assume deadlocks occur so rarely that the performance overhead of prevention, avoidance, or detection is not worth the cost.

• If a deadlock occurs, let the system freeze or require manual process termination / reboot.
Use Case: Adopted by most general-purpose operating systems, including Linux, macOS, and Windows.

3. Banker's Algorithm Numerical Example
System Setup: 5 Processes (P0 to P4) and 3 Resources: A (10 instances), B (5 instances), C (7 instances).

Step 1: Compute the Need Matrix

Formula: Need = Max - Allocation

Process Allocation (A B C) Max Demand (A B C) Calculated Need (A B C)
P00 1 07 5 37 4 3
P12 0 03 2 21 2 2
P23 0 29 0 26 0 0
P32 1 12 2 20 1 1
P40 0 24 3 34 3 1

• Total allocated: A=7, B=2, C=5.
• Available resources: [10-7, 5-2, 7-5] = [3, 3, 2].

Step 2: Safety Verification Sequence

We check if we can satisfy the Need of each process with our Available resource vector:

1. **Check P0:** Need [7, 4, 3] > Available [3, 3, 2]. Cannot allocate.
2. **Check P1:** Need [1, 2, 2] <= Available [3, 3, 2]. Allocate and run.
    • *Release resources:* New Available = [3,3,2] + [2,0,0] = [5, 3, 2].
3. **Check P2:** Need [6, 0, 0] > Available [5, 3, 2]. Cannot allocate.
4. **Check P3:** Need [0, 1, 1] <= Available [5, 3, 2]. Allocate and run.
    • *Release resources:* New Available = [5,3,2] + [2,1,1] = [7, 4, 3].
5. **Check P4:** Need [4, 3, 1] <= Available [7, 4, 3]. Allocate and run.
    • *Release resources:* New Available = [7,4,3] + [0,0,2] = [7, 4, 5].
6. **Check P0:** Need [7, 4, 3] <= Available [7, 4, 5]. Allocate and run.
    • *Release resources:* New Available = [7,4,5] + [0,1,0] = [7, 5, 5].
7. **Check P2:** Need [6, 0, 0] <= Available [7, 5, 5]. Allocate and run.
    • *Release resources:* New Available = [7,5,5] + [3,0,2] = [10, 5, 7].

System is Safe! The safe execution sequence is: <P1, P3, P4, P0, P2>.
Chapter 5

Memory Management

Contiguous allocation, paging address translation, TLB access calculations, and page replacement traces.

1. Partitioning & Fragmentations

Internal Fragmentation

Occurs when memory is allocated in **fixed-size partitions**. If a process is smaller than the partition, the leftover memory is wasted and cannot be used by other processes.

External Fragmentation

Occurs in **variable-size partitioning** when total free memory is sufficient to satisfy a request, but it is not contiguous (it's split into small holes throughout memory).
Fix: Compaction (shifting memory contents to group free spaces into a single block) or Paging.

2. Placement Strategies: First, Next, Best, Worst Fit
Example scenario: Memory partitions: 100 KB, 500 KB, 200 KB, 300 KB, 600 KB.
Process requests: 212 KB, 417 KB, 112 KB, 426 KB.

First Fit

Allocate the **first hole** that is big enough.

• 212KB → 500KB (left 288KB)
• 417KB → 600KB (left 183KB)
• 112KB → 288KB (left 176KB)
• 426KB → **Fails!** (Largest free is 300KB)

Best Fit

Allocate the **smallest hole** that is big enough. Minimizes wasted space.

• 212KB → 300KB (left 88KB)
• 417KB → 500KB (left 83KB)
• 112KB → 200KB (left 88KB)
• 426KB → 600KB (left 174KB)
• **All Fit!**

Worst Fit

Allocate the **largest hole** that is big enough. Leaves large leftover holes.

• 212KB → 600KB (left 388KB)
• 417KB → 500KB (left 83KB)
• 112KB → 388KB (left 276KB)
• 426KB → **Fails!**

3. Paging & Address Translation
Paging is a non-contiguous memory management scheme.
• **Logical Address:** Generated by the CPU. Consists of a **Page Number (p)** and a **Page Offset (d)**.
• **Physical Address:** Actual RAM address. Consists of a **Frame Number (f)** and a **Page Offset (d)**.

Logical to Physical Address Translation

1. Page size is always equal to frame size (typically 2k bytes).
2. The page offset d remains unchanged during translation.
3. The page number p is used as an index into the **Page Table**, which yields the corresponding frame number f.

Physical Address = (f * Page_Size) + d
4. Translation Lookaside Buffer (TLB)

TLB Access Math

The TLB is a fast hardware cache that stores page table translations. Let:
• h = TLB hit ratio.
• c = TLB access lookup time.
• m = Main memory access time.

EMAT (Effective Access Time) = h * (c + m) + (1 - h) * (c + 2 * m)

Why 2*m on miss? On a TLB miss, we perform one memory access to read the page table entry, and a second access to retrieve the actual data.

Numerical Example

Given hit ratio h = 90%, TLB lookup time c = 10 ns, memory access time m = 100 ns.

EMAT = 0.90 * (10 + 100) + 0.10 * (10 + 200)
     = 0.90 * 110 + 0.10 * 210
     = 99 + 21
     = 120 ns

First-In, First-Out (FIFO)

Definition: Replaces the oldest page loaded into physical memory first, utilizing a simple queue-based system.
Pros
  • Low execution overhead and simple to implement (FIFO queue of frame pointers).
  • No complex hardware access tracking structures required.
Cons
  • Belady's Anomaly: Increasing the physical frame count can counter-intuitively increase page faults.
  • Ignores how frequently or recently a page is being used, potentially replacing active pages.

Use Case: Low-cost embedded systems or devices with highly sequential memory access patterns where simplicity is preferred over optimization.

Least Recently Used (LRU)

Definition: Replaces the page in physical memory that has not been referenced for the longest duration of time.
Pros
  • Highly effective; aligns closely with the **temporal locality of reference** principle.

  • Expensive hardware support needed to track reference timestamps or stack positions on every memory request.
  • Hard to scale in pure hardware (typically approximated using reference bits, e.g. Second Chance algorithm).

Chapter 6

Disk Scheduling

HDD seek architecture, disk access times, and track seek comparison metrics.

1. HDD Architecture & Access Times
A Hard Disk Drive (HDD) consists of a stack of rotating **platters** coated with magnetic material. Electromagnetic **read-write heads** float above the platter surfaces.

HDD Track Terms

• **Tracks:** Concentric circular rings on a platter surface.
• **Sectors:** Pie-shaped slices division of tracks (typically 512 bytes).
• **Cylinder:** A set of tracks at the same distance from the spindle across all platters.

Access Time Components

• **Seek Time:** Time required to move the read-write head to the target track.
• **Rotational Latency:** Time required for the target sector to rotate under the head.
• **Transfer Time:** Time to transmit read bytes from disk to memory.

2. Numerical Disk Scheduling Traces
Track Scenario: Disk size: 200 tracks (0 to 199). Head starting position = 53.
Request queue: 98, 183, 37, 122, 14, 124, 65, 67.

First-Come, First-Served (FCFS)

Definition: Disk head services incoming requests in the exact order of their arrival in the queue, without re-ordering based on current head position.
Pros
  • Completely fair; guarantees every request is served with zero risk of starvation.
  • Incredibly simple implementation requiring only a basic FIFO queue.
Cons
  • Extremely inefficient head travel; causes high seek times and slow average response times.
  • No optimization for track proximity, causing the arm to swing erratically.

Use Case: Simple systems with light or purely sequential I/O workloads where scheduling overhead is to be minimized.

Calculated Trace:

Order: 53 → 98 → 183 → 37 → 122 → 14 → 124 → 65 → 67.
Calculation: |98 - 53| + |183 - 98| + |37 - 183| + |122 - 37| + |14 - 122| + |124 - 14| + |65 - 124| + |67 - 65|

• Total Head Movement: 640 cylinders

Shortest Seek Time First (SSTF)

Definition: Disk head always selects the next pending request that is closest to its current position, minimizing the immediate seek distance.
Pros
  • Drastically reduces average seek time compared to FCFS.
  • Increases disk throughput by minimizing unnecessary arm movements.
Cons
  • Starvation: Requests at the inner or outer extremes may wait indefinitely if active requests keep arriving near the current head position.
  • High seek time variance; response times are highly unpredictable.

Use Case: General purpose personal computers and standard workstation drives where throughput matters and extreme latency limits are not strictly enforced.

Calculated Trace:

Order: 53 → 65 → 67 → 37 → 14 → 98 → 122 → 124 → 183.
Calculation: |65 - 53| + |67 - 65| + |37 - 67| + |14 - 37| + |98 - 14| + |122 - 98| + |124 - 122| + |183 - 124|

• Total Head Movement: 236 cylinders

SCAN (Elevator Algorithm)

Definition: Disk arm moves continuously back and forth across the entire platter, servicing requests in its path. Upon reaching a boundary, it reverses direction.
Pros
  • Completely starvation-free; head reaches all tracks in a bounded sweep cycle.
  • Low seek time variance; predictable response times.
Cons
  • Inefficient boundary travel: forces the arm to travel all the way to 0 or 199 even if no requests are pending there.
  • Unequal wait times: tracks near the center are serviced twice as often as edge tracks.

Use Case: Enterprise database management systems and file servers with moderate-to-high concurrent load.

Calculated Trace (Assuming direction is toward 0 first):

Order: 53 → 37 → 14 → 0 → 65 → 67 → 98 → 122 → 124 → 183.
Calculation: (53 - 0) + (183 - 0)

• Total Head Movement: 236 cylinders

C-SCAN (Circular SCAN)

Definition: Head moves in one direction to service requests, and upon reaching the far boundary, immediately sweeps back to the beginning without servicing any requests.
Pros
  • Provides a uniform, consistent waiting time across all disk cylinders.
  • Fairer response distributions compared to standard SCAN.
Cons
  • High seek movement count on return sweeps (though modern hard disks execute this return sweep rapidly).
  • Still travels to absolute disk boundaries (0 and 199) regardless of queue density.

Use Case: Very high-load storage servers where a predictable queue execution rate is preferred over raw seek minimization.

Calculated Trace (Assuming direction is toward 199 first):

Order: 53 → 65 → 67 → 98 → 122 → 124 → 183 → 199 → 0 → 14 → 37.
Calculation: (199 - 53) + (199 - 0) + (37 - 0)

• Total Head Movement (including wrap): 382 cylinders

LOOK & C-LOOK

Definition: Optimized variants of SCAN and C-SCAN that direct the head to reverse direction immediately after servicing the last request in that direction, rather than traveling to the boundary.
Pros
  • Prevents unnecessary arm travel to the empty edges of the disk platter.
  • Saves substantial head seek distance compared to SCAN and C-SCAN.
Cons
  • Slightly increased computational overhead to continuously scan the queue for current direction boundaries.

Use Case: Default scheduling approach implemented in modern high-performance operating system block-layer drivers.

LOOK Order: 53 → 37 → 14 → 65 → 67 → 98 → 122 → 124 → 183.
LOOK Calculation: (53 - 14) + (183 - 14)

• LOOK Head Movement: 208 cylinders

Chapter 8

File Management & Security

Storage allocations, Inode file blocks, and compilation linkers/loaders.

1. File Allocation Methods

Contiguous Allocation

Each file occupies a contiguous set of blocks on the disk.
• **Pros:** Excellent read performance; simple.
• **Cons:** External fragmentation; files cannot grow easily.

Linked List Allocation

Each block contains a pointer to the next block.
• **Pros:** No external fragmentation; files can grow easily.
• **Cons:** Slow random access (requires traversing pointers); pointer overhead in blocks.

Indexed File Allocation

Brings all pointers together into a single block, called the **Index Block**. Supports fast random access and avoids external fragmentation, but incurs index block overhead for small files.

2. UNIX Inode Structure
An **Inode (Index Node)** is a data structure on a UNIX filesystem that describes a file-system object (file, directory). It stores file attributes and direct/indirect block pointers.

Block Mappings

A typical Inode has 15 pointers:
• **12 Direct Pointers:** Point directly to blocks containing file data.
• **1 Single Indirect Pointer:** Points to a block containing pointers to data blocks.
• **1 Double Indirect Pointer:** Points to a block containing pointers to indirect blocks.
• **1 Triple Indirect Pointer:** Points to a block containing pointers to double indirect blocks.

3. Linkers & Loaders

Linker

A utility program that combines multiple object files (compiled code) and libraries into a single executable file. Resolves references to external functions and variables.

Loader

An OS program that copies an executable file from disk into main memory (RAM), sets up stack and heap segments, and triggers execution at the entry point address.
