Introduction to DBMS

A Database Management System (DBMS) is a specialized system software designed to store, organize, retrieve, and manipulate data securely. It serves as an abstraction layer between physical storage and application programs or end-users.

Core Functions of a DBMS:
  • Data Storage and Buffer Management: Storing data systematically across pages on disk, using memory buffers (caching) to minimize disk I/O latency.
  • Concurrency Control: Managing simultaneous transactions using scheduling algorithms and locking protocols to prevent data corruption.
  • Security & Privilege Management: Controlling data access through granular authorization schemas (schemas, views, roles).
  • Data Integrity enforcement: Ensuring records adhere to domain constraints, referential keys, and schema rules.
  • Backup & Recovery: Logging all write operations to WAL (Write-Ahead Log) files, allowing the database engine to restore its state after hardware crashes.

DBMS Architectures

1-Tier Architecture

The client application, server database engine, and physical data live on the same local system. Primarily used for local testing or small embedded systems (e.g. SQLite embedded in mobile apps).

2-Tier Architecture

Client-Server model. The client application directly contacts the Database Server over the network, using database drivers like JDBC or ODBC to execute SQL queries.

3-Tier Architecture

Enterprise standard. The client application connects to an Application Server (Middle tier/API), which processes the business logic and queries the backend Database Server.

Distributed Deployments

Homogeneous Distributed DBMS

All physical database servers run the exact same DBMS software engine (e.g. all nodes are PostgreSQL). They share schema rules, making queries and distributed joins simple to coordinate.

Heterogeneous Distributed DBMS

Different physical nodes run different database engines (e.g. some run Oracle, others MySQL). Requires specialized middleware translation layers to reconcile schema mapping and query operations.

Codd's 12 Rules for Relational Databases

Dr. E.F. Codd established 12 rules (numbered 1 through 12, plus Rule 0) that define a true Relational Database Management System (RDBMS):

Rule # Rule Name Description
Rule 0 Foundation Rule The system must manage databases entirely through its relational capabilities.
Rule 1 Information Rule All data (including metadata) must be represented as values in table columns and rows.
Rule 2 Guaranteed Access Rule Every data item is logically accessible using table name + primary key + column name.
Rule 3 Systematic Null Treatment Nulls must represent missing/inapplicable info in a systematic way, distinct from 0 or blank.
Rule 4 Dynamic Online Catalog Database schema description must be stored in tables queryable using SQL.
Rule 5 Comprehensive Sublanguage Must support a single language (like SQL) covering schema definition, manipulation, integrity, and transactions.
Rule 6 View Updating Rule All database views that are theoretically updatable must be updatable by the system.
Rule 7 High-Level Insert/Update/Delete Must support multi-row insert, update, and delete statements.
Rule 8 Physical Data Independence Changes to physical storage structures (disks, file paths) must not affect application code.
Rule 9 Logical Data Independence Changes to conceptual schemas (splitting tables, adding fields) must not impact view applications.
Rule 10 Integrity Independence Integrity constraints (PK, FK) must be stored in catalog metadata, not embedded in application code.
Rule 11 Distribution Independence The database must function normally even if data is moved across different physical servers.
Rule 12 Nonsubversion Rule Low-level access routes must not bypass schema security or integrity constraints.
Entity-Relationship
ER Model & Mapping

The Entity-Relationship (ER) model is a conceptual, abstract database design framework representing entities, attributes, and relationships. It is mapped to a relational schema during the implementation phase.

Entities & Entity Sets

Strong Entity Set

Can exist independently of other entities. Must have a primary key (composed of key attributes) to uniquely identify each instance.

Symbol: Single RectangleExample: Customer (CustID)

Weak Entity Set

Cannot exist without a corresponding strong entity set (its owner entity). Does not have a primary key. It is identified using the owner's primary key combined with its own partial key / discriminator attribute.

Symbol: Double RectangleExample: Payment (PaymentNo, discriminator)

Cardinality & Structural Constraints
Participation Constraints (min cardinality):
• Total Participation (Double Line): Every entity in the set must participate in at least one relationship instance. (e.g., every Employee must belong to a Department).
• Partial Participation (Single Line): Entities can exist without participating in the relationship. (e.g., not every Employee manages a Department).

Min-Max Notation (min, max):
• Placed on the line connecting the entity set and relationship.
• min = 0 indicates partial participation. min ≥ 1 indicates total participation.
• max represents the cardinality ratio limit (e.g., 1 or N).
Generalization, Specialization, Aggregation & Inheritance

Generalization & Specialization

Generalization: Bottom-up approach where common attributes of lower-level entities are combined into a high-level parent entity.
Specialization: Top-down approach where a parent entity is decomposed into lower-level sub-entities based on specialized attributes.

Aggregation

Abstraction where a relationship set (and participating entities) is enclosed in a box and treated as a single higher-level entity, allowing it to have relationship links to other entities.

Types of Inheritance

  • Attribute Inheritance: Lower-level entities inherit all attributes of the parent entity automatically.
  • Relationship Inheritance: Lower-level entities participate in all relationships that the parent entity participates in.
  • Overriding Inheritance: A sub-entity can redefine a parent attribute or relationship for its specific type.
ER-to-Relational Mapping Algorithm

Step-by-step logic to map conceptual ER diagrams into database tables:

1
Strong Entity
Create a table. Simple attributes become columns. Composite attributes are flattened into individual columns (e.g. Street, City columns). Key attribute becomes the Primary Key.
2
Weak Entity
Create a table. Add all attributes of the weak entity. Put the Primary Key of the owner entity as a Foreign Key. The Primary Key is a composite key: {Owner_PK, Discriminator}.
3
1:1 Relationships
Based on participation constraints:
• Both Total: Merge the two entity sets into a single table.
• One Total, One Partial: Take the PK from the partial side and place it as a Foreign Key in the total participation side's table.
• Both Partial: Keep separate tables and add a Foreign Key to either side (preferably the side with fewer null rows).
4
1:N / N:1 Relationships
Do not create a new table. Take the Primary Key from the "1" side and place it as a Foreign Key in the table representing the "N" side (to avoid null values).
5
M:N Relationships
Must create a new table. The columns of the new table are the primary keys of both participating entities (each acting as a foreign key). The primary key of this new table is the composite key {PK1, PK2}.
6
Multivalued Attribute
Create a new table. Place the parent entity's Primary Key (as FK) and the multivalued attribute column in this table. The composite PK is {Parent_PK, Attribute}.

SQL Masterclass

Structured Query Language (SQL) is the standard declarative implementation language for database query operations. This section covers production-grade queries, syntax, and structures.

DDL & DML Syntax

Creating tables with constraints

Common field constraints include PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, CHECK, and DEFAULT.

CREATE TABLE Employees ( EmpID INT PRIMARY KEY, Name VARCHAR(100) NOT NULL, Email VARCHAR(100) UNIQUE, Salary DECIMAL(10,2) CHECK (Salary > 0), DeptID INT, JoinDate DATE DEFAULT CURRENT_DATE, CONSTRAINT fk_dept FOREIGN KEY (DeptID) REFERENCES Departments(Id) ON DELETE SET NULL );
SQL Joins Deep-Dive

Consider two tables: Employees (ID, Name, ManagerID) and Departments (ID, DeptName).

Self Join Query
Cross Join Query

Self Join (Table references itself)

Typically used for hierarchical columns in a single table, like employee-manager relations.

-- Find employees who earn more than their direct manager SELECT e.Name AS Employee, m.Name AS Manager FROM Employees e INNER JOIN Employees m ON e.ManagerID = m.EmpID WHERE e.Salary > m.Salary;

Cross Join (Cartesian Product)

Returns the cartesian product of rows from both tables without matching conditions.

-- Generate all possible combinations of sizes and colors SELECT s.SizeName, c.ColorName FROM Sizes s CROSS JOIN Colors c;
Aggregations & Group By
Aggregate functions (COUNT, SUM, AVG, MIN, MAX) perform calculations on a set of values.
• GROUP BY: Groups rows sharing identical values in summary columns.
• HAVING: Filters grouped results. Equivalent to WHERE but runs after values are aggregated.
-- Get departments with more than 5 employees and an average salary above 60000 SELECT DeptID, COUNT(*) AS EmpCount, AVG(Salary) AS AvgSalary FROM Employees GROUP BY DeptID HAVING COUNT(*) > 5 AND AVG(Salary) > 60000;
Nested & Subqueries

Correlated Subquery (EXISTS)

The inner query executes repeatedly — once for each row evaluated by the outer query.

-- Find departments with active employees SELECT d.DeptName FROM Departments d WHERE EXISTS ( SELECT 1 FROM Employees e WHERE e.DeptID = d.ID );

Common Table Expressions (CTEs)

The WITH clause creates a temporary named result set, making complex queries readable.

-- Filter employees earning above average WITH AvgSal AS ( SELECT AVG(Salary) AS avg_s FROM Employees ) SELECT Name, Salary FROM Employees, AvgSal WHERE Salary > avg_s;

Set Operations & Views

UNION vs UNION ALL

UNION combines query results and deduplicates rows (costly sort). UNION ALL merges row sets without removing duplicates, making it much faster.

Views & Indexes

Non-Relational
NoSQL Databases

NoSQL (Not Only SQL) databases are built for horizontal scalability, unstructured schemas, and high write throughput, departing from the rigid ACID constraints of relational systems.

Why NoSQL?
Traditional RDBMS scales vertically (faster CPU, more RAM), which has physical scaling limits and high hardware costs. NoSQL is designed to scale horizontally (adding cheap server nodes in a distributed cluster).
• Dynamic Schemas: Allows storing unstructured document records (JSON) without running migrations.
• Low Latency: Uses simple query patterns (e.g. key-value hash lookups) to bypass slow relational joins.
SQL vs NoSQL Comparison
Dimension SQL Databases (RDBMS) NoSQL Databases
Schema Strict, predefined tabular schema. Changes require DDL migration locks. Dynamic / Schema-less. Records can store custom dynamic columns.
Scaling Vertical (Scale Up). Horizontal (Scale Out). Shards data partitions across multiple nodes.
Joins Supported natively. Executed horizontally. Not supported. Data is denormalized in documents to avoid joins.
Transactions Strict ACID properties. Absolute consistency. BASE model. Prioritizes availability; eventual consistency.
The 4 Types of NoSQL Databases

1. Document Database

Stores records as documents (JSON/BSON). Documents can nest objects and arrays inside themselves.

Use Cases: E-commerce catalogs, Content Management Systems (CMS), user profile configurations.
MongoDBCouchDB

2. Key-Value Store

Stores simple key-to-value associations. Extremely fast because it behaves as a giant hash table in RAM.

Use Cases: Real-time caching, user session states, global leaderboards.

3. Column-Family Store

Stores columns of data together on disk, rather than rows. Highly optimized for mass writes across columns.

Use Cases: IoT sensor metrics, time-series telemetry data, event logs.

4. Graph Database

Stores data nodes and directed edges (relationships) natively, bypassing join layers for deep networks.

Database Design
Normalization & Anomaly Prevention

Normalization is the mathematical process of decomposing tables to reduce data redundancy, eliminate data anomalies, and preserve functional dependencies.

Database Anomalies Illustrated

Suppose we store student and course info in one unnormalized table:

Table: StudentCourse StudentID | Name | CourseID | CourseName | DeptHead ----------+---------+----------+------------+--------- 101 | Alice | CS10 | DBMS | Dr. Smith 101 | Alice | CS20 | Networks | Dr. Smith 102 | Bob | CS10 | DBMS | Dr. Smith

Insertion Anomaly

We cannot insert a new course (e.g. CS30, OS under Dr. Jones) unless we have at least one student enrolled in it, because StudentID is part of the primary key.

Deletion Anomaly

If Bob (StudentID 102) drops CS10 and we delete his row, we completely lose the details that CS10 is called DBMS and managed by Dr. Smith if he was the only enrollee.

Update Anomaly

If the department head for CS changes from Dr. Smith to Dr. Alan, we have to update multiple rows. If we miss even one row, the database state becomes inconsistent.

Finding Candidate Keys Algorithm
Algorithm:
1. Identify the set of all attributes in relation R.
2. Find attributes that never appear on the right-hand side (RHS) of any Functional Dependency (FD). These attributes must be part of every candidate key.
3. Compute the closure of this core attribute set. If the closure covers all attributes of R, it is the unique Candidate Key.
4. If not, add other attributes one by one and calculate closures until you find minimal sets that determine R.
Armstrong's Axioms in Detail
Axiom Rule Mathematical Definition Explanation / Example
Reflexivity If Y ⊆ X, then X → Y If X contains Y, then X determines Y. e.g. {ID, Name} → {Name}
Augmentation If X → Y, then XZ → YZ Adding attributes to both sides preserves dependency. e.g. If ID → Name, then {ID, Code} → {Name, Code}
Transitivity If X → Y and Y → Z, then X → Z Chain rule. e.g. If StudentID → ZipCode and ZipCode → City, then StudentID → City
Union (Secondary) If X → Y and X → Z, then X → YZ If X determines two fields separately, it determines both.
Decomposition (Secondary) If X → YZ, then X → Y and X → Z We can split the RHS attributes into separate dependencies.
Normal Forms (1NF - 5NF)
1NF, 2NF, 3NF
BCNF, 4NF, 5NF
Normal Form Core Requirement Dependency Avoidance
1NF (First Normal Form) All column values must be atomic (no multi-valued attributes, no repeating groups). Removes duplicate records and structural repeating nesting.
2NF (Second Normal Form) Must satisfy 1NF, and have no Partial Dependency. No non-prime attribute should depend on a proper subset of a candidate key.
3NF (Third Normal Form) Must satisfy 2NF, and have no Transitive Dependency. For every non-trivial FD X → Y, either X is a superkey or Y is a prime attribute.
Normal Form Core Requirement Details / Focus
BCNF (Boyce-Codd Normal Form) Stricter version of 3NF. For every non-trivial FD X → Y, X must be a superkey. Eliminates cases where a prime attribute is determined by a non-prime attribute.
4NF (Fourth Normal Form) Must satisfy BCNF and contain no Multi-Valued Dependencies (MVD). Prevents independent 1-to-many relationships from creating duplicate records in one table.
5NF (Fifth Normal Form) Must satisfy 4NF and have no Join Dependency. Ensures a table cannot be decomposed and reconstructed without losing data.
Decomposition Properties

1. Lossless Join Decomposition

Ensures that when relation R is split into R1 and R2, joining them back (R1 ⋈ R2) reconstructs R exactly without inserting spurious tuples.
Condition: R1 ∩ R2 ≠ ∅ and either (R1 ∩ R2) → R1 or (R1 ∩ R2) → R2.

2. Dependency Preserving Decomposition

Guarantees that all functional dependencies of the original relation can be validated inside the individual tables without performing joins.
Condition: (F1 ∪ F2)+ = F+.

Denormalization
Denormalization is an optimization technique where redundant data is intentionally added back to a normalized schema.
• Why? Excessively normalized databases require complex, slow multi-table joins.
• Tradeoff: Dramatically increases read speed but adds update overhead and storage requirements. Ideal for read-heavy analytical dashboards (OLAP), not transaction engines (OLTP).
Concurrency
Transactions, Recovery & Deadlocks

A transaction is a logical unit of work. To maintain database consistency, transactions must adhere to ACID properties, and concurrent operations must be serializable.

ACID vs BASE
ACID (Strict Consistency)
BASE (Eventual Consistency)
ACID Properties:
• Atomicity: All-or-nothing execution. Rollback on failure. (Managed by Transaction Manager)
• Consistency: The database transitions only between valid states conforming to rules. (Managed by Application Programmer)
• Isolation: Concurrent transactions run independently without interference. (Managed by Concurrency Control Manager)
• Durability: Once committed, changes survive system crashes. (Managed by Recovery Manager)
BASE Model (NoSQL systems):
• Basically Available: The system responds to requests, even if some nodes are down.
• Soft State: The state of the data can drift or change over time without user interaction.
• Eventual Consistency: Data copies across global servers will sync up and become identical over time.
CAP Theorem
States that a distributed data store can simultaneously provide at most two of the following three guarantees:
1. Consistency (C): Every read receives the most recent write or an error.
2. Availability (A): Every non-failing node returns a response for every request.
3. Partition Tolerance (P): The system continues to operate despite arbitrary communication network splits.
Tradeoff: When a network partition (P) occurs, a database must choose either Consistency (CP) or Availability (AP).
Concurrency Problems

Dirty Read

Transaction T1 reads data modified by T2. T2 then aborts and rolls back, leaving T1 with invalid, uncommitted data.

Lost Update

T1 and T2 read the same value, modify it independently, and write it back. One update overwrites the other.

Non-Repeatable Read

T1 reads a value. T2 updates that value and commits. T1 reads the value again and gets a different result.

Schedules & Serializability
Serial Schedule: Transactions execute one after another without interleaving. Always consistent. Non-Serial Schedule: Operations interleave. Improves performance but can lead to conflicts. Conflict Serializable: A schedule that can be transformed into a serial schedule by swapping non-conflicting adjacent operations. View Serializable: A weaker check than conflict serializability, verifying identical read/write flow states.
Recoverability of Schedules
R
Recoverable Schedule
T2 reads a value written by T1. T2 commits only after T1 commits. If T1 aborts, T2 can also safely abort.
C
Cascadeless Schedule
T2 is only allowed to read values written by T1 after T1 has committed. Prevents cascading rollbacks where one transaction failure forces aborting multiple active ones.
S
Strict Schedule
A transaction cannot read or write a data item written by another transaction until that transaction has committed or aborted. Prevents dirty writes.

Two-Phase Locking (2PL) Rules

Guarantees conflict serializability but does not prevent deadlocks:

  • Growing Phase: Transaction can acquire locks but cannot release any.
  • Shrinking Phase: Transaction can release locks but cannot acquire new ones.
  • Strict 2PL: All exclusive (X) locks held by a transaction must be kept until the transaction commits.

Database Recovery Techniques

Log-Based Recovery

Uses a sequential write-ahead log file to recover data:

  • Deferred Update (No-Undo/Redo): Changes written to disk only after commit. On failure, apply REDO. No UNDO is ever needed.
  • Immediate Update (Undo/Redo): Changes can be written to disk before commit. On failure, apply UNDO for uncommitted, and REDO for committed transactions.

Shadow Paging

Alternative recovery scheme that avoids logs. Keeps two page tables: a Current Page Table and a Shadow Page Table. On commit, the current pointer overwrites the shadow pointer.

Deadlock Prevention Schemes
Wait-Die Scheme
Wound-Wait Scheme
Wait-Die Scheme (Non-preemptive):
• If an older transaction requests a resource held by a younger one, the older is allowed to wait.
• If a younger transaction requests a resource held by an older one, the younger is killed (dies) and restarted.
Wound-Wait Scheme (Preemptive):
• If an older transaction requests a resource held by a younger one, the older pre-empts (wounds) the younger, forcing it to abort and roll back.
• If a younger transaction requests a resource held by an older one, the younger is allowed to wait.
Optimization
Indexing & Physical Design

Indexing is an essential data structure technique designed to speed up search operations by reducing the number of physical disk block accesses required to retrieve rows.

Indexing Classifications
Index Type Key Order Alignment Index density Description
Primary Index Ordered key matches table structure. Usually Sparse Defined on the primary key of an ordered data file. Only one primary index can exist.
Clustering Index Non-key field ordered matches table. Sparse Defined on a non-key ordered field (e.g., grouping employees by Department ID).
Secondary Index Table physical order is independent of index key. Dense Defined on unordered fields (e.g., creating an index on email field). Multiple secondary indexes are allowed.
Index Dense vs Sparse

Dense Index

An index record is created for every single search key value in the database table. Faster lookup, but requires significantly more memory.

Sparse Index

An index record is created only for some search keys (typically one per data block). Requires less memory, but requires traversing the physical block to find the exact row.

B-Tree vs B+ Tree Node Layout

Database engines use B+ Trees for relational indexes due to their physical layout advantages:

Feature B-Tree B+ Tree
Data pointers Stored in both leaf and non-leaf nodes. Stored only in leaf nodes. Non-leaf nodes only store search key values to route queries.
Leaf linking Leaf nodes are not linked together. Leaf nodes are connected sequentially in a doubly linked list.
Range Queries Slow. Requires executing in-order traversal (up and down the tree). Fast. Locate the starting key in a leaf, then traverse the linked list.
Internal Node capacity Low, because data pointers occupy space in internal nodes. High. Can store many child pointers, leading to a much wider fan-out and shorter tree.
Preparation
Interview Q&A & Tricks

Curated DBMS interview prep, core conceptual focus points, and common database interview cheats.

Top 10 Frequently Asked Interview Questions

Q1: Explain the difference between Clustered and Non-Clustered index.

A clustered index dictates the physical order in which rows are stored on disk. Because the data rows can only be sorted in one order, there can be at most one clustered index per table. A non-clustered index has a separate structure that contains search keys and pointers to the actual data rows. You can create multiple non-clustered indexes on a table.

Q2: Why are B+ Trees preferred over Binary Search Trees (BST) for disk storage?

BSTs are deep and narrow, causing a high number of node travels. Since database pages live on disk, reading a node requires a disk block I/O. B+ Trees have a high fan-out (many children per node), making the tree short and wide. A search requires only 3-4 block reads even for millions of records, reducing I/O bottleneck.

Q3: What is the difference between 3NF and BCNF?

BCNF is a stricter form of 3NF. In 3NF, for any functional dependency X → Y, either X is a superkey, or Y is a prime attribute. In BCNF, X must be a superkey. BCNF eliminates partial dependencies where a prime attribute is determined by a non-prime attribute.

Q4: Explain the difference between a Conflict and View Serializable schedule.

A schedule is conflict serializable if it can be reduced to a serial schedule by swapping non-conflicting adjacent operations. A schedule is view serializable if it is view equivalent to some serial schedule. Every conflict serializable schedule is view serializable, but not vice versa (e.g., schedules with blind writes).

Q5: What are data anomalies and how does normalization solve them?

Anomalies are side-effects of data redundancy in unnormalized tables: Insertion Anomaly (cannot add data without unrelated parent keys), Deletion Anomaly (losing data as a side-effect of deletion), and Update Anomaly (inconsistencies when duplicate records are not updated simultaneously). Normalization splits tables so that each fact is stored in exactly one place.

Q6: Explain the difference between optimistic and pessimistic concurrency control.

Pessimistic concurrency control locks resources before modifications occur, assuming conflicts are highly likely. Optimistic concurrency control allows transactions to execute without locking, but checks for conflicts during the commit phase (using validation steps); it rolls back the transaction if a conflict occurred, assuming conflicts are rare.

Q7: What is write-ahead logging (WAL)?

WAL is a database recovery protocol stating that all database modifications must be written to a persistent log file (on disk) before the actual changes are applied to the physical database pages. This guarantees durability and recoverability in case of power failure.

Q8: Explain the difference between 2-Tier and 3-Tier architecture.

In a 2-Tier architecture, the client application runs queries directly on the database engine. In a 3-Tier architecture, the client communicates with an application server (API/backend), which performs validation/business logic and then queries the database engine, offering better scalability and security.

Q9: Why does Natural Join remove duplicate columns while Cartesian Product keeps them?

Natural Join matches columns by name and merges them into one logical attribute to simplify relationships. Cartesian Product does not analyze column names; it simply pairs every attribute from Table A with every attribute from Table B, resulting in duplicate columns containing prefix names.

Q10: What are weak entity sets, and how do they map to schemas?

Weak entity sets are entities that cannot be uniquely identified using only their own attributes and depend on a strong parent entity. They map to a separate table where the columns are the weak entity's attributes, plus the parent's Primary Key. The primary key of the weak table is a composite key: {Parent_PK, Discriminator}.

Interview Tricks & Common Confusions

Thomas Write Rule Edge Case

An optimization on the Timestamp Concurrency protocol. If transaction T wants to write a data item, and the write timestamp is older than the current write timestamp of the item, instead of rolling back T, it simply discards the write because it was overwritten anyway. Only works if isolation levels allow.

Null vs Empty Values

In databases, NULL represents the absence of data, whereas an empty string or 0 represents actual entered data. Standard SQL logical operations treat nulls using three-valued logic (True, False, Unknown).
