System Design Mastery

A complete reference built from your course notes, whiteboard sessions, and case study images. Every concept defined. Every trade-off explained. Every case study dissected.

Three Levels of Design

High-Level Design (HLD): How systems interact — servers, databases, APIs, protocols, data flow.

Low-Level Design (LLD): Classes, functions, data structures, algorithms, code-level decisions.

System Principles: Scalability, fault tolerance, consistency, availability — the why behind every decision.

What Makes a Good System

A production-grade system must be:

• Scalable — handles growth without degrading
• Fault tolerant — survives component failure
• Extensible — easy to add features without rewrites
• Observable — logs, metrics, tracing, alerting
• Decoupled — services don't know each other's internals

Fundamentals

Scaling

The ability of a system to handle growing amounts of work by adding resources. Two directions: up or out.

Definition
Scalability means your system can grow to handle more users, requests, or data without degrading performance. The two core strategies have completely different tradeoffs — most mature systems use both.
Vertical vs Horizontal

Vertical Scaling (Scale Up)

Add more CPU, RAM, or faster disk to the same machine. No code changes needed. Fast inter-process communication. But has a hard hardware limit and is a single point of failure.

Example: Upgrading from 16 GB RAM → 256 GB RAM on your DB server.

Horizontal Scaling (Scale Out)

Add more machines to distribute load. Theoretically unlimited. Requires load balancers, introduces network latency, and makes data consistency harder.

Example: Adding 20 more API servers behind a load balancer when traffic doubles.

Vertical Wins
  • Simple — no application changes
  • Fast IPC (no network hop)
  • Strong consistency
  • Great for small teams / early stage
Horizontal Wins
  • No single point of failure
  • Virtually unlimited scale
  • Resilient — one node dies, others continue
  • Cost-effective at scale (commodity hardware)
Hybrid Strategy (Real World)
Best practice: Start vertical (simpler, fewer moving parts). Once you hit hardware limits or need fault tolerance, scale horizontally. Most companies run vertically-scaled nodes that are themselves part of a horizontally scaled cluster. Cassandra: each node is powerful, and there are 100 of them.
Pre-processing at Non-Peak Hours

Batch Work During Off-Hours

Heavy computations (generating recommendations, pre-building news feeds) are done at low-traffic windows (e.g., 3am). Results are cached and served instantly during peak. Instagram pre-computes your feed rather than building it live on every request — this is why your feed loads in milliseconds even though it aggregates posts from hundreds of people.

Key Takeaways

Why Caching Matters

RAM (100 ns) is 300,000× faster than disk (30 ms). If your hot data fits in RAM, never let it hit disk. This is why Redis is so transformative.

Why CDNs Matter

Cross-continent = 150ms. CDN edge node = 10–20ms. A 10× improvement in perceived load time for static assets.

Why SSD Matters

SSD eliminates disk seek: 10ms → 0.1ms. Random reads improve by 100×. Modern databases default to SSD-backed storage for this reason.

Minimize Network Calls

Each datacenter round trip = 0.5ms. 100 micro-calls = 50ms of pure network overhead. Batch your calls aggressively.

Theory

CAP Theorem

In any distributed system, you can guarantee at most 2 of 3: Consistency, Availability, Partition Tolerance.

Eric Brewer's CAP theorem (2000): a distributed data store can only guarantee two of the three properties simultaneously. Since network partitions are inevitable in real distributed systems, you are always effectively choosing between Consistency and Availability during a partition.

Consistency

Every read receives the most recent write or an error. All nodes see the same data at the same time.

Availability

Every request receives a response — not necessarily the latest data. The system stays up even when some nodes fail.

Partition Tolerance

The system operates even when network messages between nodes are dropped. Required in all real distributed systems.

CP vs AP — The Real Choice

CP Systems

Return errors during network partitions rather than stale data. Strong consistency guaranteed.

Examples: HBase, ZooKeeper, MongoDB (default), etcd, Redis (single master)

Use when: Financial transactions, inventory management, anything where stale data causes real harm.

AP Systems

Return potentially stale data during partitions but never return errors. Eventual consistency.

Examples: Cassandra, DynamoDB, CouchDB, DNS

Use when: Social feeds, product catalogs, user profiles — anything tolerant of momentary staleness.

ACID vs BASE

ACID (SQL Databases)

Atomicity — all or nothing
Consistency — valid state before and after every transaction
Isolation — concurrent transactions don't interfere
Durability — committed data survives crashes

BASE (NoSQL Databases)

Basically Available — system is available most of the time
Soft state — state may change without input (replication lag)
Eventual consistency — data will converge to consistency eventually

2-Phase Commit (2PC)
Distributed transaction protocol. Phase 1 (Prepare): coordinator sends PREPARE to all nodes; nodes lock resources and reply OK. Phase 2 (Commit): if all replied OK, coordinator sends COMMIT; nodes commit and release locks. If any node fails in Phase 1, coordinator sends ROLLBACK. Problem: coordinator is a SPOF; protocol blocks if coordinator crashes after Phase 1 but before Phase 2.
Core Concepts

Load Balancing

Distributes incoming traffic across multiple servers to maximize throughput, minimize latency, and ensure no single server is overwhelmed.

Definition
Load Balancing is the process of distributing network or application traffic across a cluster of servers. It improves responsiveness and increases availability of applications.
Algorithms

Round Robin

Requests distributed sequentially. Simple, works well when servers have equal capacity and requests are similar in weight.

Weighted Round Robin

Servers with higher capacity get more requests. Useful when servers have different hardware specs.

Least Connections

New requests go to the server with fewest active connections. Best for variable-length requests.

IP Hash

Client's IP is hashed to always route to the same server. Good for session stickiness without session storage.

L4 vs L7 Load Balancing

L4 (Transport Layer)

Routes based on IP and TCP/UDP port. Fast but no content inspection. Can't route based on URL path or HTTP headers.

L7 (Application Layer)

Routes based on HTTP headers, URL, cookies, content. Slower but powerful — route /api/* to API servers, /static/* to CDN.

Health Checks: Load balancers continuously ping backend servers. If a server fails to respond, it's removed from the rotation automatically — this is how zero-downtime failover works.
Core Concepts

Consistent Hashing

A hashing technique that minimizes key remapping when nodes are added or removed from a distributed system.

The Problem
With simple modular hashing (key % N servers), adding or removing a server causes almost all keys to remap — catastrophic for cache systems. Consistent hashing solves this: only K/N keys need remapping when a node is added/removed (K = keys, N = nodes).
How It Works

The Hash Ring

Arrange all possible hash values in a circle (0 to 2³²). Each server is placed at a point on the ring via its hash. Each key is also hashed and assigned to the nearest server clockwise. When a server is added, only keys between the new server and its predecessor need to move.

Virtual Nodes: Each physical server is placed at multiple points on the ring (virtual nodes). This ensures even distribution even with heterogeneous servers, and further reduces hotspots. Used by Cassandra, DynamoDB, and Riak.
Core Concepts

Caching

Storing frequently accessed data in fast storage (RAM) to avoid repeated expensive computations or database queries.

Definition
A cache is a high-speed data storage layer that stores a subset of data, typically transient, so that future requests for that data are served faster. Cache hit = data found. Cache miss = must fetch from source.
Cache Strategies

Write-Through

Write to cache AND database simultaneously. Data always consistent but higher write latency. Good for read-heavy workloads.

Write-Back (Write-Behind)

Write to cache only; async flush to DB later. Low write latency but risk of data loss on cache crash.

Cache-Aside (Lazy Loading)

App checks cache first. On miss, app fetches from DB and populates cache. Most common pattern. Cache only holds accessed data.

Read-Through

Cache sits in front of DB. On miss, cache fetches from DB automatically. Transparent to application.

Eviction Policies
LRU — Least Recently Used
LFU — Least Frequently Used
FIFO — First In First Out
Random Replacement
TTL — Time To Live
Redis vs Memcached: Redis supports rich data structures (lists, sets, sorted sets, hashes), persistence, pub/sub, and Lua scripting. Memcached is simpler but faster for pure key-value. Redis is the industry standard for most use cases.
Core Concepts

Message Queues

Asynchronous communication buffers that decouple producers from consumers, enabling independent scaling and fault tolerance.

Definition
A message queue stores messages sent by producers until consumers are ready to process them. This decouples services — the producer doesn't need to wait for the consumer to be available. Classic examples: RabbitMQ, Amazon SQS, ActiveMQ.
Advantages
  • Decouples producer and consumer
  • Handles traffic spikes gracefully
  • Messages persisted until consumed
  • Retry logic built in
Trade-offs
  • Added latency (async)
  • Ordering not guaranteed (usually)
  • Message duplication possible
  • Operational complexity
Use cases: Order processing, email sending, image resizing, video transcoding, payment processing — any task that can be done asynchronously and shouldn't block the user response.
Core Concepts

Pub / Sub

A messaging pattern where publishers send events to topics, and multiple subscribers independently consume those events.

Definition
In Pub/Sub, publishers emit events without knowing who will consume them. Subscribers express interest in topics and receive all matching events. Unlike message queues (one consumer per message), pub/sub delivers each event to all subscribers. Examples: Apache Kafka, Google Pub/Sub, Redis Pub/Sub.
Pub/Sub vs Message Queue

Message Queue

One message → one consumer. Message deleted after consumed. Used for task distribution — work is shared among workers.

Pub/Sub

One message → all subscribers. Each subscriber gets their own copy. Used for event broadcasting — notifications, analytics, cache invalidation.

Kafka: Kafka is technically a distributed log that supports both patterns. Messages are retained (not deleted after consumption) and consumers track their own offset. This enables replay, multiple consumer groups, and extremely high throughput (millions of events/second).
Core Concepts

Database Sharding

Horizontally partitioning a database into smaller, independent pieces (shards) each stored on a separate server.

Definition
Sharding splits a large database into smaller shards, each responsible for a subset of data. Each shard is an independent database that can be queried independently. Enables horizontal scaling of databases beyond what any single machine can handle.
Sharding Strategies

Hash-Based Sharding

Hash the shard key (e.g., user_id % N). Even distribution but makes range queries hard and resharding painful.

Range-Based Sharding

Shard by key ranges (A-M, N-Z). Easy range queries but can create hotspots if data isn't uniformly distributed.

Geographic Sharding

Shard by user region. Keeps data close to users, helps with data residency regulations.

Directory-Based Sharding

A lookup table maps keys to shards. Flexible but the lookup table becomes a bottleneck and SPOF.

⚠️ Hotspot Problem: If one shard gets disproportionate traffic (celebrity user, trending content), it becomes overloaded. Mitigate with consistent hashing, virtual nodes, or application-level load awareness.
Core Concepts

Replication

Maintaining copies of data across multiple nodes for fault tolerance, read scaling, and geographic distribution.

Definition
Replication keeps identical copies of data on multiple servers. If one server fails, others serve requests. Also allows reads to be distributed across replicas for higher throughput.
Replication Patterns

Primary-Replica (Master-Slave)

All writes go to primary. Replicas sync async and serve reads. Simple, but primary is a SPOF. Failover requires promoting a replica.

Multi-Primary (Multi-Master)

Multiple nodes accept writes. Higher availability but introduces write conflicts — requires conflict resolution strategy.

Sync vs Async Replication

Synchronous

Write doesn't return success until all replicas confirm. Zero data loss but higher latency. Used in financial systems.

Asynchronous

Write returns immediately after primary confirms. Replicas catch up later. Low latency but potential data loss on primary crash.

Core Concepts

Content Delivery Network (CDN)

A geographically distributed network of servers that caches and serves content from locations close to users.

Definition
A CDN caches static content (images, CSS, JS, videos) at edge servers worldwide. User requests are routed to the nearest edge node — reducing latency from 150ms (cross-continent) to 10–20ms. Examples: Cloudflare, AWS CloudFront, Akamai, Fastly.
Push vs Pull CDN

Push CDN

You proactively push content to edge nodes. Full control over what's cached. Good for known, large assets (videos). Must manage cache invalidation yourself.

Pull CDN

Edge fetches from origin on first miss, caches with TTL. Simpler. Cache fills organically with popular content. Risk of cache miss storms on new content.

CDN for dynamic content: Modern CDNs can also accelerate dynamic content via persistent connections, TCP optimization, and smart routing — not just static files. Cloudflare Workers lets you run code at the edge.
Architecture

Microservices

Architectural style where a large application is built as a suite of small, independently deployable services.

Definition
In microservices, each service owns a specific business capability, has its own database, and communicates via well-defined APIs (REST, gRPC, events). Services are independently deployable, scalable, and replaceable.
Advantages
  • Independent deployment and scaling
  • Technology diversity per service
  • Fault isolation
  • Easier to understand small codebases
  • Parallel development by multiple teams
Trade-offs
  • Network latency between services
  • Distributed system complexity
  • Data consistency challenges
  • Operational overhead (many services)
  • Hard to trace requests across services
vs Monolith: Start with a monolith. Extract microservices when a specific component has different scaling requirements or team boundaries emerge. Don't start with microservices on day one — the overhead kills small teams.
Architecture

Event-Driven Architecture

A design pattern where services communicate by producing and consuming events, enabling loose coupling and high scalability.

Definition
In event-driven architecture, services emit events when something happens (order placed, user signed up). Other services subscribe to relevant events and react. The event broker (Kafka, SNS) is the central nervous system.

Reliability

Single Point of Failure

A component whose failure causes the entire system to fail. Eliminating SPOFs is the foundation of high availability.

Definition
A Single Point of Failure (SPOF) is any component that, if it fails, takes down the entire system. High-availability design is fundamentally the exercise of identifying and eliminating SPOFs through redundancy.
Common SPOFs and Solutions

Single DB Server

Solution: Primary-replica setup with automatic failover. Use managed services (RDS Multi-AZ) that handle failover automatically.

Single Load Balancer

Solution: Active-active or active-passive LB pairs. DNS failover. Use cloud-managed LBs (AWS ALB) with built-in redundancy.

Single Data Center

Solution: Multi-region deployment. Active-active or active-passive across regions. DNS routing via latency/health checks.

Single Network Provider

Solution: Multi-homing — connect to multiple ISPs. BGP routing for automatic failover.

Five Nines (99.999%): Only 5.26 minutes of downtime per year. Achieving this requires redundancy at every layer — compute, network, storage, DNS — with automated health checks and failover.
Core Concepts

API Design

Principles and patterns for designing robust, scalable, and developer-friendly application programming interfaces.

REST vs GraphQL vs gRPC

REST

Resource-based, HTTP verbs, stateless, JSON. Universal client support. Over-fetching/under-fetching is the main pain. Use for public APIs.

GraphQL

Client specifies exact data needed. Single endpoint. No over/under-fetching. Complex queries can be expensive server-side. Great for mobile clients.

gRPC

Binary protocol (Protocol Buffers). 5–10× faster than REST. Strongly typed. Bi-directional streaming. Used for internal microservice communication.

Rate Limiting

Rate Limiting Algorithms

Token Bucket: Tokens added at fixed rate; each request consumes a token. Allows bursts.

Leaky Bucket: Requests processed at fixed rate regardless of input. Smooths bursts.

Fixed Window Counter: Count requests per time window. Simple but vulnerable to boundary attacks.

Sliding Window Log: Track timestamps of each request. Precise but memory-intensive.

Idempotency: Design mutating API endpoints to be idempotent — calling them multiple times produces the same result. Use idempotency keys (unique request IDs) to deduplicate retried requests. Critical for payment APIs.
Deep Dive

Database Internals

Understanding how databases store, index, and retrieve data — essential for making informed schema and query decisions.

B-Tree Index
The most common index structure. A balanced tree where all leaves are at the same depth. Supports equality and range queries. Read-optimized: O(log n) lookups. Used by PostgreSQL, MySQL (InnoDB), SQLite. B+ Tree variant: data only in leaf nodes, leaves linked for efficient range scans.
LSM Tree (Log-Structured Merge-Tree)
Write-optimized structure. Writes go to in-memory buffer (MemTable), flushed to sorted files (SSTables) on disk. Background compaction merges SSTables. Reads may check multiple levels — mitigated by Bloom filters. Used by: Cassandra, RocksDB, LevelDB, HBase. Trade-off: faster writes than B-tree, slower reads without optimization.
WAL — Write-Ahead Log
Before modifying data, the database writes the change to a WAL (append-only log). If the system crashes, the WAL is replayed to restore consistency. This is how ACID durability is implemented without flushing every write to disk immediately.
Core Concepts

SQL vs NoSQL

Choosing the right database type is one of the most impactful architectural decisions. Neither is universally better.

Comparison

SQL (Relational)

Examples: PostgreSQL, MySQL, SQLite, SQL Server

• Structured schema, enforced relationships
• ACID transactions
• Powerful JOIN queries
• Vertical scaling primarily
• Best for: financial data, inventory, user accounts, anything with complex relationships

NoSQL

Examples: MongoDB, Cassandra, DynamoDB, Redis

• Flexible schema, horizontal scaling
• Eventual consistency (usually)
• Designed for specific access patterns
• Massive scale
• Best for: social graphs, product catalogs, real-time analytics, IoT data, caching

NoSQL Types

Key-Value

Redis, DynamoDB. Fastest. Perfect for sessions, caches, leaderboards.

Document

MongoDB, Firestore. JSON documents. Great for content, user profiles, catalogs.

Wide-Column

Cassandra, HBase. Optimized for time-series, write-heavy workloads at massive scale.

Graph

Neo4j, Amazon Neptune. Nodes and edges. Best for social networks, recommendations, fraud detection.

Systems

Location-Based Systems

Designing systems that store, query, and serve geospatial data efficiently — used in Uber, Tinder, Google Maps, and Yelp.

Core Challenge
Finding all points within a radius of a given location efficiently. Naive approach (scan all records and compute distance) doesn't scale. Solution: spatial indexing — partition space so only nearby candidates are checked.
Spatial Indexing Techniques

Geohash

Encodes (lat, lng) as a base-32 string. Nearby locations share a common prefix. Easy to implement, supported natively by Redis GEOSEARCH. Limitations at cell boundaries.

Quadtree

Recursively subdivides space into 4 quadrants. Dynamically adapts to data density. Used internally by many mapping systems.

S2 Library (Google)

Maps sphere to 1D curve using Hilbert curve. Preserves locality. Used by Google Maps, Uber, and many others.

R-Tree

Index for rectangles and polygons. Standard in PostGIS/PostgreSQL spatial extensions. Good for arbitrary shapes.

Feed Generation: Push vs Pull

Fan-out on Write (Push)

When user posts, pre-populate feed for all followers. Feed reads are instant. But celebrity with 100M followers = 100M writes per post. Instagram uses hybrid — fan-out for regular users, pull for celebrities.

Fan-out on Read (Pull)

Build feed at read time by fetching posts from all followed users. No write amplification. But high read latency. Works well for low-follower accounts.

Case Study

Design Netflix

A global video streaming service with 250M subscribers. Core challenges: content delivery at global scale, personalization, encoding pipeline.

Requirements

Functional

Stream videos globally · Personalized recommendations · Multiple profiles per account · Download for offline · Continue watching across devices

Netflix Open Connect CDN
Netflix operates its own CDN called Open Connect. They place servers (OCAs) directly inside ISP networks globally. During off-peak hours, Netflix proactively pushes popular content to OCAs close to viewers. Result: 95%+ of traffic served from local ISP caches — dramatically reducing costs and improving quality.
Recommendations

Collaborative Filtering

Users similar to you watched X → recommend X. Powered by offline ML training on massive viewing history. Netflix A/B tests every UI element — even the thumbnail shown for each title is personalized per user to maximize click rate.

Chaos Engineering: Netflix created Chaos Monkey — a tool that randomly kills production services to ensure the system handles failures gracefully. This is how they achieve extreme resilience: by practicing failure constantly.

Steps

4. Peak QPS: 2-3× average QPS
5. Storage: Size of one unit × count per day × retention period
6. Bandwidth: Avg request size × QPS

Interview

Interview Tips & Framework

A structured approach to system design interviews. The difference between senior and staff-level candidates is almost entirely in how they structure their thinking.

The 5-Step Framework

1. Clarify Requirements (5 min)

Don't assume. Ask: Who are the users? What's the scale? Read vs write heavy? Any specific constraints? Latency requirements? Global or regional? Active users vs registered?

2. Estimate Scale (3 min)

QPS, storage, bandwidth. This drives your architectural choices. A system serving 1K users vs 1B users is completely different. Do this out loud — show your math.

3. High-Level Design (10 min)

Draw the major components: clients, load balancers, API servers, databases, caches, queues, CDN. Don't go deep yet — get the full picture down first.

4. Deep Dive (15 min)

The interviewer will guide you to the interesting part. Common deep dives: database schema, feed generation algorithm, consistency model, handling failures.

5. Trade-offs (5 min)

Every decision has a cost. Acknowledge them: "We chose Cassandra for writes but this makes JOINs impossible." Senior engineers always discuss what they gave up.

Wrap Up

Discuss bottlenecks, monitoring strategy, how the system evolves with 10× more traffic, any open issues you'd address with more time.

Do's and Don'ts
Think out loud. Interviewers want to see your thought process, not just the answer. A great wrong answer with good reasoning beats a lucky right answer with no explanation.
Drive the conversation. Don't wait to be asked. Lead with: "I'm going to start with requirements, then estimate scale, then sketch the HLD. Does that sound good?"
Don't jump to solutions. Designing before clarifying requirements is the #1 interview mistake. You'll solve the wrong problem and waste everyone's time.
Don't say "use Kafka" without reasoning. Every technology choice needs a why. "Kafka because it handles high throughput and message replay" beats "Kafka" alone.
Acknowledge trade-offs explicitly. "We could use PostgreSQL here for strong consistency, but at 1M writes/sec we'd need heavy sharding. Cassandra trades consistency for write throughput — that's acceptable here because..."
