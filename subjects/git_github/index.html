Foundations

Version Control Systems (VCS)

Why tracking source history is critical for clean, collaborative software development.

Definition

A Version Control System (VCS) is a software utility that tracks changes made to files (usually source code) over time. It allows developers to review modifications, revert files to previous states, compare revisions, and collaborate without overwriting each other's work.

Track History

Maintains a timeline (log) of every addition, modification, and deletion. You can trace who changed what line of code, when, and why.

Collaborate

Allows multiple developers to work simultaneously on the same project codebase without blocking one another.

Why VCS Exists

Without a VCS, developers are forced to manually copy folders (e.g. project_v1_final_v2), coordinate changes via chat, and manually resolve conflicting modifications. A VCS automates this tracking, providing a safety net to undo mistakes instantly.

Distributed vs Centralized VCS

Modern version control is divided into two primary architectures:

FeatureCentralized VCS (e.g. SVN)Distributed VCS (e.g. Git)
History LocationSingle central server onlyEvery developer has a full copy of the history locally
Offline OperationsCannot commit, view log, or diff offlineCan perform almost all operations offline
SpeedSlower (requires remote network connections)Near-instant (local filesystem reads/writes)
ResilienceSingle point of failure (server crash = loss of history)Highly resilient (every local copy is a backup)

Analogy: The Document Track Changes

Analogy

A VCS is like the **"Version History"** panel in Google Docs. Instead of saving a new document copy every time you write a paragraph, the system records only the incremental edits (diffs). If a paragraph gets accidentally deleted, you can scroll back in time and restore that exact block.

Git vs GitHub

The critical distinction between the local tool and the cloud hosting service.

Git (The Tool)

Git is a free, open-source, fast, and scalable distributed version control system that runs locally on your computer's command terminal. It creates and tracks local history logs.

GitHub (The Platform)

GitHub is a commercial, web-based hosting platform that stores your Git repositories in the cloud. It provides features like Pull Requests, issue tracking, user access management, and visual collaboration tools.

Key Distinctions

DimensionGitGitHub
NatureLocal CLI software programCloud hosted web platform (SaaS)
InstallationRequired on developer's local machineNo installation (accessed via browser)
NetworkRuns entirely offlineRequires internet connection to sync/push
Key CompetitorsMercurial, SVN, PerforceGitLab, Bitbucket, AWS CodeCommit
Real-world Analogy

Think of **Git** as the camera on your phone that takes pictures (commits) and stores them in your local gallery. Think of **GitHub** as Instagram — a public/private web platform where you upload (push) those pictures to share and collaborate with others.

Installation & Configuration

Setting up the command line, verifying versions, and writing global settings.

Verifying Installation

Before using Git, check if it is already installed in your system's path by opening a terminal and running:

git --version
Output example

git version 2.39.3 (Apple Git-145)

Setting Up User Configuration

When you commit changes, Git stamps them with your name and email. These must be set globally using the git config command:

# Set your global username
git config --global user.name "John Doe"

# Set your global email address
git config --global user.email "johndoe@gmail.com"

Verifying Configurations

To view all configured settings and trace where they are saved, run:

git config --list
Global configurations are written to a file named .gitconfig located in your user's primary root directory (e.g. ~/.gitconfig). You can open and edit it directly.
Local Git

Git Lifecycle & File Status

Understanding the four states of files and how Git tracks modifications.

As you edit, add, or delete files in your local project workspace, Git classifies every file into one of four distinct states:

1. Untracked

Files that exist in your working directory but have never been added to Git's database index. Git ignores them.

2. Modified

Files already tracked by Git that have changes in your working directory, but the changes aren't staged yet.

3. Staged

Files that have been flagged as ready to be committed. Staging prepares the snapshot of changes.

4. Unmodified

Files that are committed in Git's database and match the local working directory exactly.

Repository & Cloning

Creating a local folder copy of remote GitHub repositories.

Repository (Repo)

A Repository is a database folder containing all your project files and their complete version history logs (the metadata is stored inside a hidden folder called .git).

Cloning a Remote Repository

To duplicate an existing remote repository hosted on GitHub down to your local machine, use the git clone command along with the HTTPS or SSH link:

git clone https://github.com/username/my-project.git

This command automatically does the following:

  • Creates a new folder matching the repository name.
  • Initializes a local Git database inside it.
  • Fetches all the history branches and commits.
  • Checks out the default branch (usually main or master).
  • Configures the remote source address as origin.

Basic Terminal Operations

cd [dir]Change DirectoryNavigates into the specified folder.

ls -aList AllLists all files, including hidden files (like the .git folder).
Local Git

Staging Area & Adding Files

Preparing changes for a commit snapshot by moving them to the staging index.

Staging Area

The Staging Area (or index) is a conceptual middle ground in Git. It is a file that tracks changes that are ready to be snapshotted in the next commit. This allows developers to stage only a subset of their modifications instead of committing everything at once.

Staging Specific Files

To stage changes in a specific file, use the git add command:

git add index.html

Staging All Changes

To stage all new, modified, and deleted files in the entire project folder in one command, use the dot wildcard:

git add .

Analogy: Packing boxes for moving

Analogy

Imagine packing boxes to move houses. Your **Working Directory** is your room with items strewn about. Running **git add** is like picking up specific items and loading them into a box. The box represents the **Staging Area**. You haven't sealed it or shipped it yet; you are just choosing what goes in the box.

Local Git

Committing Changes

Creating a permanent, signed snapshot of staged files in the Git database.

Commit

A Commit is a permanent snapshot of your staged changes saved directly to the local repository history. Each commit has a unique alphanumeric SHA-1 hash identifier, an author stamp, and a descriptive commit message.

Running a Commit

To commit your staged changes, use the -m flag to supply a short, descriptive message in the terminal:

git commit -m "Add responsive mobile navigation layout"

Analogy: Saving a Game State

Analogy

A commit is like creating a **Save State** (checkpoint) in a video game. If you fail a boss fight or make a mistake, you can reload that exact save file to restore your character's state. The commits form a linear history of save states.

Common Commit Mistakes

Mistake
  • Writing generic messages like "Fixed stuff" or "Update".
  • Committing massive, unrelated changes at once, making bugs hard to isolate.
Correct Pattern
  • Use imperative mood (e.g. "Fix CSS bug" instead of "Fixed CSS bug").
  • Commit early and commit often, grouping changes atomically.
Local Git

Initializing & Remote Repositories

Creating a repository from scratch locally and binding it to a remote GitHub URL.

Creating a New Local Repository

If you have an existing project folder locally that isn't under version control, you can initialize a Git database inside it using:

# Initialize local repository
git init

This creates a hidden folder named .git in the root directory.

Binding a Remote Source (GitHub)

To sync this local repository with a remote repository you created on GitHub, add a remote binding address named origin:

# Bind local repository to remote GitHub URL
git remote add origin https://github.com/username/my-repo.git

# Verify connected remote URLs
git remote -v

Renaming the Default Branch

By default, git init historically names the default branch master. To rename it to the modern standard main, use:

git branch -M main
Remote & Branching

Pushing Code to GitHub

Uploading committed snapshots from your local machine to the remote repository.

Push

Pushing is the process of uploading your local repository commit snapshots to a remote repository server like GitHub, making your local history available to collaborators.

Setting Up Upstream & Pushing

The first time you push code from a new local branch, you must bind it to the remote branch using the upstream flag (-u):

# Push and set upstream destination
git push -u origin main

For subsequent pushes, once the upstream is configured, you can run the simplified command:

git push

Analogy: Publishing a Draft

Analogy

Writing code is like writing a blog post. Saving drafts is like committing locally on your laptop (private). **Pushing** is like hitting "Publish" or syncing it to your cloud drive, making it public to the internet so others can read and edit it.

Branching & Navigation

Creating parallel lines of development to build features in isolation.

Branch

A Branch in Git is a lightweight, moveable pointer to a specific commit. By default, projects start with a default branch (typically main). Creating a new branch creates a separate pathway of history so you can build features or fix bugs without affecting the stable production code.

Merging & Pull Requests

Integrating isolated branches and requesting code reviews on GitHub.

Comparing Changes

To view the difference in code content between two branches, commits, or staging areas before merging, use:

git diff feature-branch main

Merging Locally

To integrate changes from a feature branch into your active branch (e.g., merging feature-login into main), run:

# 1. Switch to the recipient branch (main)
git checkout main

# 2. Merge changes from the target branch
git merge feature-login

Pull Requests (PRs)

Pull Request

A Pull Request (PR) is a collaboration feature on GitHub that notifies team members that you've pushed changes to a branch and would like them to review, comment on, and merge those changes into the main repository.

Fast-Forward vs Three-Way Merge

Fast-Forward Merge

If the recipient branch hasn't diverged (no new commits were made to main while you worked on the feature), Git simply moves the branch pointer forward. No merge commit is created.

Three-Way Merge (3-Way)

If both branches have diverged (new commits were made on both branches), Git performs a three-way merge using a common ancestor and creates a new, dedicated merge commit.

Remote & Branching

Pulling & Syncing

Updating your local repository to match remote updates on GitHub.

Pull

Pulling fetches the latest commits from the remote repository and immediately integrates (merges) them into your active local branch.

Running a Pull

# Fetch and merge remote changes
git pull origin main

Git Fetch vs Git Pull

It is critical to distinguish between fetch and pull:

git fetch
Downloads remote changes to database only
Safe (doesn't modify working files)
git pull
Downloads remote changes + immediately merges
Active (can cause merge conflicts)
Under the hood, running git pull is exactly equivalent to running git fetch followed immediately by git merge.
Advanced Operations

Resolving Merge Conflicts

What happens when Git cannot merge automatically, and how to resolve it manually.

Merge Conflict

A Merge Conflict is an event that occurs when Git tries to merge two branches that have conflicting changes in the same line of the same file, or when a file has been deleted in one branch but modified in the other. Git cannot decide which version to keep and pauses the merge.

How Git Flags Conflicts

When a conflict occurs, Git marks the conflict boundaries directly in the affected files using conflict markers:

<<<<<<< HEAD
This is my local code change on main branch.
=======
This is the incoming code change from feature branch.
>>>>>>> feature-branch

Step-by-Step Conflict Resolution

  1. Open the conflicted files in an editor (VS Code highlights these sections).
  2. Inspect the code and manually choose which changes to keep, removing the markers (<<<<<<<, =======, >>>>>>>).
  3. Stage the resolved files: git add [filename].
  4. Finalize the merge by committing: git commit -m "Resolve merge conflict".
Advanced Operations

Undoing Changes

Resetting states, restoring working directories, and reverting history.

Case 1: Staged changes (Unstaging files)

If you ran git add but decided you aren't ready to commit those files yet, unstage them using:

# Unstage a specific file
git reset index.html

# Unstage all staged changes
git reset

Case 2: Committed changes (Undoing commits)

If you committed changes locally but want to pull that commit back, you can reset. Git defines three modes of resetting, referencing HEAD~1 (one commit back from active branch point):

Reset ModeImpact on Working DirectoryImpact on Staging AreaLog History
Soft (--soft)Keeps all modified files intactKeeps all staged changes intactMoves commit pointer back
Mixed (Default)Keeps all modified files intactUnstages all changesMoves commit pointer back
Hard (--hard)Deletes all working changesDeletes all staged changesMoves commit pointer back

Running a Hard Reset

# WARNING: This destroys all uncommitted local modifications permanently
git reset --hard HEAD~1

# Reset to a specific commit hash in history
git reset --hard a1b2c3d
Warning

Never run a hard reset on commits that have already been pushed to a shared remote repository. It rewrites log history, causing severe sync issues for other team members.

Forking Repositories

Creating independent copies of remote projects on your GitHub profile.

Fork

A Fork is a new remote repository that shares code and visibility settings with the original upstream project, but is copied entirely onto your personal GitHub account profile. It allows you to freely experiment and push changes without affecting the parent project.

Fork vs Clone

DimensionForkClone
LocationCreated on GitHub's remote serverCreated on your local computer's drive
RelationshipTethered to the original project (supports PRs)Direct downstream mirror copy of a target remote
AccessNo write access required on the original projectRequires write permissions to push back to remote
Use Case

Forking is the primary workflow for **Open-Source Contributions**. You fork a public library, clone your fork locally, make improvements, commit and push to your fork, and then submit a **Pull Request** to merge those changes back into the original creator's repository.
