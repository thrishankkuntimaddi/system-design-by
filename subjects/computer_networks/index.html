What is a Computer Network

Where the whole subject starts: machines, wires, and an agreement about how to talk to each other.

Definition

A computer network is a collection of computers connected together so they can exchange data. The internet is the largest example — a network of networks spanning the planet.

A playful backronym sometimes used for "computer": Commonly Oriented Machine Particularly used for Training, Education and Research — not the literal etymology, but a useful way to remember that a computer at its core is a general-purpose machine for processing information, and a network is what happens when many of those machines are wired together to share it.

A short history — how the internet started

Before the internet, computers were isolated islands. The story of how that changed:

ARPA founded
→
ARPANET (research network)
→
TCP adopted
→
More hosts join
→
World Wide Web
  • ARPA (Advanced Research Projects Agency) — a U.S. government agency that funded early research into how computers could communicate with each other over long distances.
  • ARPANET — the first real packet-switched network ARPA built, originally linking research institutions including MIT, Stanford, UCLA, and the University of Utah, with early links partly carried over satellite connections.
  • TCP (Transmission Control Protocol) — the rule-set invented to let independent, dissimilar computer networks reliably exchange data with each other. This is the single biggest unlock: once everyone agreed on TCP, networks could interconnect instead of staying isolated.
  • Over years, more institutions, companies, and eventually homes connected — the network of networks became the Internet.
  • WWW (World Wide Web) — not the same thing as the internet. The internet is the physical/logical network of connected computers; the Web is a collection of linked documents (pages) that live on top of that network, accessed via HTTP.
Common confusion

"Internet" and "Web" are often used interchangeably but aren't the same: the internet is the infrastructure (cables, routers, protocols); the Web is one application that runs on that infrastructure, alongside others like email, file transfer, and video calls.

How a request actually travels

At the simplest level, when Computer A wants to talk to Computer B — possibly on the other side of the planet — the message doesn't teleport. It is broken down, addressed, and physically carried across a chain of intermediate devices until it reaches the destination, and a reply makes the same kind of journey back.

Scenario — typing google.com

You type google.com in a browser and hit enter → your computer turns that into a request → the request is sent through your router, your ISP, and a chain of other networks → it reaches Google's server → the server sends back a response (the webpage) → your browser renders it. This client → server round trip is the basic shape of almost everything that happens on a network.

Protocols & Core Terminology

The vocabulary you need before any of the layered models make sense.

Definition

A protocol is a set of rules that defines how data is formatted, transmitted, and interpreted between devices. Just like two people need a shared language to have a conversation, two computers need a shared protocol to exchange meaningful data.

Protocols aren't invented by a single company — they're standardized, often region-by-region and globally, by bodies like the Internet Society, with the actual technical specifications published as RFCs (Request For Comments) through the RFC Editor. An RFC is essentially the "spec document" engineers refer back to when implementing a protocol correctly.

Client–Server model

The dominant pattern for how applications talk over a network:

Client
request →
Server
Client
← response

  • Client — initiates the request (your browser, your phone app).
  • Server — a machine that's always listening, waiting to respond to requests (identified by an IP address).

Casting modes: Unicast, Broadcast, Multicast

How data is addressed and distributed across the network determines its casting mode, operating at both Layer 2 (MAC) and Layer 3 (IP):

Unicast (One-to-One)

Data is sent from a single sender to a single specific receiver. The packet contains one destination IP/MAC address. Used for web browsing, emails, and direct file transfers.

Broadcast (One-to-All)

Data is sent from a single sender to all hosts on the same local network subnet. Packets are addressed to a special broadcast address (e.g. 255.255.255.255). Routers block broadcasts to keep traffic localized. Used for ARP resolution and DHCP discovery.

Multicast (One-to-Many)

Data is sent from a single sender to a specific group of interested receivers who have subscribed to a multicast group IP (ranges 224.0.0.0 to 239.255.255.255). Switches and routers replicate the packets only to ports with active subscribers. Used for live IPTV, video conferencing, and routing protocol updates.

Real-world Analogy

Unicast is sending a private letter to a specific home. Broadcast is shouting through a megaphone in a neighborhood square where everyone hears you. Multicast is sending a magazine only to subscribers of a particular interest club.

Circuit Switching vs Packet Switching

The two fundamental paradigms of moving data through a network mesh:

Circuit Switching

A dedicated physical communication path (circuit) is established between the sender and receiver before any data is sent. The capacity of the path is reserved and cannot be shared. If no data is sent, the resource is wasted.

Packet Switching

Data is broken into smaller chunks called packets, each containing a header with routing info. Packets travel independently through the network, sharing paths dynamically (statistical multiplexing) and being queued at routers (store-and-forward).

Postal vs Phone Analogy

Circuit Switching is like making a phone call: the line is reserved just for your voice, and no one else can use it until you hang up. Packet Switching is like sending a multi-chapter book through the mail: each page is placed in an envelope and mailed separately. They might take different routes and arrive out of order, but they are reassembled at the destination.

IP Addresses

Definition

An IP address is a numeric label assigned to every device on a network, used to identify and locate it — conceptually, the "phone book entry" of the internet.

Every server you talk to is identified by an IP address, even though you usually type a domain name instead (DNS bridges that gap — covered later).

Try it

On macOS/Linux, run curl ifconfig.me -s in a terminal — it returns your machine's public IP address as seen from the internet.

ISPs and the chain to your device

Your ISP (Internet Service Provider — e.g. Airtel, Jio, Comcast) is the company that actually connects your home or office to the wider internet.

  1. The ISP gives you a modem/router, which is assigned a global (public) IP address — visible to the rest of the internet.
  2. Inside your home, the router connects to internal devices (laptop, phone, smart TV).
  3. Each of those internal devices gets a private/local IP address, automatically handed out by DHCP (Dynamic Host Configuration Protocol).

Ports

Definition

A port number identifies a specific application or service running on a device — the IP address gets you to the right machine, the port gets you to the right program on that machine.

  • Ports are 16-bit numbers → range from 0 to 2¹⁶−1, i.e. roughly 65,535 available ports.
  • 0–1023 — well-known / reserved ports (e.g. HTTP = 80, HTTPS = 443, MongoDB = 27017).
  • 1024–49151 — registered ports, used by specific named applications.
  • 49152–65535 — dynamic/ephemeral ports, free for short-lived use (e.g. the temporary port your browser tab uses for one connection).

Network Delay, Throughput & Goodput

When measuring network performance, bandwidth alone is insufficient. We must track how long data takes to travel, how much arrival time varies, and how much actual data is successfully processed.

Latency (Delay)

The total time it takes for a data packet to travel from the source to the destination host. Latency is the sum of four distinct components: Nodal Delay = Processing Delay + Queuing Delay + Transmission Delay + Propagation Delay.

Jitter

The variation or fluctuation in packet arrival times (packet delay variation). High jitter causes packets to arrive out of order or clumped together, resulting in choppy audio/video during VoIP calls or video streams.

Throughput vs Goodput

Throughput is the total rate at which bits are sent over the physical link per unit time (including TCP/IP headers, metadata, and retransmissions). Goodput is the rate of actual useful application-level payload data successfully delivered (excluding all protocol header overhead and duplicate packets).

Throughput vs Goodput example

If you download a 10MB file and due to a poor connection 2MB of packets are corrupted and retransmitted, your link's **throughput** counts all 12MB of data sent. However, the **goodput** only counts the 10MB of successfully reassembled file payload delivered to your browser.

The Four Sources of Packet Delay (Nodal Delay)

A packet experiences different delays at each router hop along its path:

1. Processing Delay (d_proc)

The time a router takes to inspect a packet's header, check for bit-level errors using checksums, and determine the next-hop interface. Usually microseconds or less.

2. Queuing Delay (d_queue)

The time a packet spends waiting in the router's buffer queues before being transmitted. Depends heavily on network congestion and the traffic intensity ($L \cdot a / R$ where $L$ is packet length, $a$ is arrival rate, $R$ is transmission rate). Can range from microseconds to seconds.

3. Transmission Delay (d_trans)

The time taken to push all the packet's bits onto the physical wire. Formula: d_trans = L / R (where L is packet length in bits, and R is the transmission rate of the link in bps). Depends on packet size and wire speed.

4. Propagation Delay (d_prop)

The time it takes for a single bit to travel physically from one end of the link to the other at the speed of light in the physical medium. Formula: d_prop = d / s (where d is physical distance, and s is propagation speed, roughly $2 \times 10^8$ m/s in optical fiber). Depends only on distance and medium, not packet size.

Memory Aid

Remember the delay sources top-to-bottom: People Quickly Take Pathways — Processing, Queuing, Transmission, Propagation.

Quality of Service (QoS)

A suite of technologies (like traffic shaping, packet tagging, and priority queuing) used to manage traffic congestion. QoS ensures high-priority, latency-sensitive packets (like live voice/video) are processed ahead of non-critical traffic (like bulk file downloads).

Guided vs unguided media

Guided

Data travels along a physically defined path — copper or fiber cable. Faster and more reliable because the path is fixed and shielded from interference.

Unguided

Data travels through open space with no single fixed path — radio waves. Used by Wi-Fi and Bluetooth (short range) and 3G/4G/LTE/5G (long range).

How continents are physically connected

Even satellite and radio aside, the backbone of global internet traffic is physical undersea cables — optical fiber laid across ocean floors connecting continents. You can see the actual global map of these at submarinecablemap.com, maintained by TeleGeography — it shows hundreds of named cables (2Africa, AEConnect-1, etc.) physically linking landing points across every continent.

Physical cables remain faster and more reliable than wireless long-haul links, which is why almost all intercontinental internet traffic — even from "wireless" companies — ultimately rides on undersea fiber.

Network Topologies

The five classic shapes you can wire a network into — and exactly why each one wins or loses in practice.

A topology is the arrangement/layout of how nodes (computers, devices) are connected to each other in a network. The choice of topology directly affects cost, reliability, and how a failure behaves.

1. Bus Topology

All devices are connected to a single shared central cable (the "bus" / backbone). Every device taps into this one line to send and receive data.

Pros
  • Very cheap to set up — minimal cabling
  • Easy to extend by adding a new tap
  • Good for small, simple networks
Cons
  • If the single backbone cable breaks anywhere, the entire network is split or goes down
  • All devices share the same line → unnecessary collisions/calls when many devices transmit at once
  • Hard to troubleshoot — a fault anywhere affects everyone

2. Ring Topology

Each device connects to exactly two neighbors, forming a closed loop. Data travels around the ring, hop by hop, until it reaches its destination.

Pros
  • Predictable, orderly data flow (token-passing variants avoid collisions entirely)
  • Performs consistently even with many devices, unlike a shared bus
Cons
  • A single broken link can disrupt the entire ring (unless dual-ring redundancy is used)
  • Adding/removing a device usually requires briefly breaking the ring
  • One slow or failed device can hold up the whole loop

3. Star Topology

Every device connects individually to one central device (a hub or switch). All communication passes through this central point.

Pros
  • One device failing doesn't affect any other — easy isolation of faults
  • Easy to add/remove devices without disrupting the network
  • Centralized management and monitoring
Cons
  • If the central device (hub/switch) fails, the entire network goes down — single point of failure
  • More cabling required than bus topology
In practice

Most modern home and office LANs are star topologies — every device cables back to a central Wi-Fi router or Ethernet switch.

4. Tree Topology (Bus + Star hybrid)

A hierarchical structure: multiple star networks are connected together along a common bus backbone — essentially "stars hanging off a bus," forming a parent-child structure of hubs.

Pros
  • Scales well — easy to add new branches/departments without redesigning the whole network
  • Fault isolation within a branch, similar to star
  • Matches how real organizations are structured (HQ → departments → individual machines)
Cons
  • Still depends on the backbone bus — if that breaks, branches get isolated from each other
  • More complex to wire and maintain than pure star or pure bus

5. Mesh Topology

Every device is connected directly to every other device. Maximum redundancy — there are multiple paths between any two nodes.

Pros
  • Extremely fault-tolerant — many alternate paths if one link fails
  • No single point of failure
  • High performance, no shared-bus contention
Cons
  • Expensive — cabling grows roughly with n² as devices are added
  • Scalability issues — becomes physically impractical past a small number of nodes

LAN, MAN, WAN & Network Structure

How small local networks chain together to form the internet — and the analogy that makes layering click.

Network scale classifications

TermFull nameScaleTypical medium
LANLocal Area NetworkA single home, office, or buildingEthernet cables, Wi-Fi
MANMetropolitan Area NetworkAcross a cityFiber backbones connecting multiple LANs
WANWide Area NetworkAcross countries / continentsSONET (Synchronous Optical Network), leased lines, satellite

Frame Relay is a WAN protocol historically used to connect LANs across a wide area cost-effectively, by packaging data into variable-length "frames" sent over a shared carrier network — a precursor to many of today's WAN/VPN technologies.

Big picture

The internet is simply the collection of all interconnected LANs, MANs, and WANs across the world — your home LAN connects to your ISP's MAN, which peers into global WAN backbones, which connect to other countries' networks.

Modem / Router — the bridge between worlds

A modem converts digital signals (what your computer understands) into analog signals (what travels over telephone/cable lines) and vice versa — the literal translator between your home network and your ISP's infrastructure. A router is the device that decides where to forward data packets next, both within your home network and out to the internet (in home setups, the ISP-provided device is usually a combined modem+router).

Tier 1 vs Tier 2 ISPs

  • Tier 1 ISPs — own global backbone infrastructure and don't pay anyone for transit (e.g. TATA Communications). They peer directly with each other.
  • Tier 2 ISPs — regional/national providers (e.g. Airtel, Jio) that buy transit from Tier 1s to reach the rest of the internet, while also peering with each other locally.

The Amazon delivery analogy

This is the mental model that makes the entire layered approach (OSI/TCP-IP) click: shipping a physical package mirrors exactly how data is "shipped" across a network.

Notice each stage only cares about its own job: Amazon doesn't drive the truck, the delivery company doesn't know what's inside the box, and the truck driver doesn't know who the recipient is beyond an address label. This is exactly the idea of layering — each network layer only does its specific job and hands the package to the next layer, ignorant of what the layers above or below are doing with the contents.

Application layer connection

In this analogy, "placing the order" and "receiving the package" are the parts you directly interact with — this maps to the Application Layer, the only layer that's visible to the end user. Everything else (the courier routing, the trucks, the roads) happens invisibly underneath, just like the lower OSI layers.

The OSI Model — Overview

Open Systems Interconnection: the standard mental map for every layer of networking, top to bottom.

Definition

The OSI model is a conceptual, standardized framework that describes how data moves through a network in 7 distinct layers, each with a specific responsibility. It's "open" because it's vendor-neutral — any system following it can interoperate with any other.

Data physically only exists as electrical/light/radio signals — the OSI model is the human-friendly abstraction layered on top so engineers can reason about "what's happening" without tracking every electron. Each layer talks logically to its counterpart layer on the other machine, but physically only ever talks to the layer directly above/below it on the same machine.

Models · OSI Layer 7

Layer 7 — Application Layer

The gateway between end-user applications and the underlying network interface.

Definition

The Application Layer is the top layer of the OSI model, representing the user-facing software environment that initiates or receives network requests. It directly supports software applications (browsers, email clients, file sharing tools) by providing a suite of standardized protocols to exchange data.

PDU: Data / Message
Focus: User Experience & APIs
Hardware: Gateway / Host CPU

Core Purpose & Function

This layer does not refer to the actual software application itself (like Google Chrome or Outlook), but to the underlying protocols that those applications use. It acts as the direct interface between the program and the network stack, negotiating services such as resource discovery, protocol selection, and user identification.

Key Responsibilities

  • Application Interface: Exposing network services (like sockets or HTTP request headers) directly to user-facing software.
  • Resource Discovery: Identifying communication partners, verifying network availability, and coordinating authority to connect.
  • Service Standardization: Ensuring that application-level exchanges (like sending an email or downloading a file) follow strict, structured patterns across different operating systems.

Analogy: The Amazon Order Interface

Amazon Analogy

In the physical delivery model, the Application Layer maps to the Amazon web interface or mobile app where you place the order. You specify what you want, who you are, and where it should go. You don't care how the package is shipped, which trucks are used, or how it is routed — you only deal with the high-level request itself.

Pros & Cons of Application Layer Abstraction

Pros
  • High-level developer ease: no need to manage hardware interfaces or routing logic.
  • Standardized, interoperable interfaces across different platforms.
  • Allows rich user interaction and visual formatting.
Cons
  • Significant performance overhead due to heavy text parsing and payload headers.
  • Highly vulnerable to security exploits (SQL injection, XSS, API spoofing).
  • Inefficient for latency-critical, raw bitstream needs.
Models · OSI Layer 6

Layer 6 — Presentation Layer

The data translator, responsible for formatting, encrypting, and compressing payloads.

Definition

The Presentation Layer acts as the translation assistant for the network. It takes structured data from the Application Layer and formats, serializes, encrypts, or compresses it into a standardized syntax that can be sent over the wire, and performs the reverse operation for incoming streams.

PDU: Data / Syntax
Focus: Data Formats & Security
Hardware: CPU / Crypto Accelerators

Core Purpose & Function

Different operating systems represent data differently (e.g., EBCDIC vs ASCII text, or little-endian vs big-endian integers). The Presentation Layer resolves these syntax discrepancies, ensuring that the destination system can correctly interpret the data payload, regardless of its local architecture.

Key Responsibilities

  • Data Translation: Serializing in-memory programming objects into common serialization formats (like JSON, XML, or binary Protocol Buffers) and decoding them at the receiver.
  • Encryption & Decryption: Securing data for transit by applying cryptographic protocols (like SSL/TLS) to encrypt sensitive application payloads.
  • Data Compression: Compressing data to minimize the total number of bytes sent over the network, maximizing bandwidth efficiency.

Common Standards & Protocols

JSON / XML / YAMLSerialization SyntaxConverts software structures into readable text files.
Protobuf / FlatBuffersBinary SerializationConverts data structures into compact binary format for gRPC/speed.

Analogy: The Package Wrapping & Gift Box

Amazon Analogy

The Presentation Layer corresponds to the packaging and wrapping of your order. If you order a fragile item, it is wrapped in bubble wrap (compression) and sealed in an opaque box (encryption) to protect it from outside inspection and damage. It is also labeled in a standardized language/format that the courier network understands.

Key Responsibilities Matrix

Strengths
  • Ensures cross-platform interoperability (Windows can talk to Linux/Mac).
  • Centralizes cryptography, keeping application logic clean.
  • Reduces data volume through transparent compression.
Weaknesses
  • Encryption/Decryption is computationally heavy and can introduce latency bottlenecks.
  • Bad formatting configurations (like character set mismatches) cause data corruption (mojibake).
Models · OSI Layer 5

Layer 5 — Session Layer

The connection manager, coordinating and keeping conversations in sync.

Definition

The Session Layer manages the dialog between two host computers. It establishes, coordinates, maintains, and terminates connections (sessions) between local and remote applications, keeping track of authorization, session tokens, and connection synchronization.

PDU: Data / Session
Focus: Dialogue & Checkpoints
Hardware: CPU / Auth Servers

Core Purpose & Function

While the lower layers manage the delivery of raw packets or segments, the Session Layer is the high-level manager that understands the *lifetime* of the conversation. It handles user authentication, session token mapping, and coordinates which party is allowed to talk (dialogue control).

Key Responsibilities

  • Session Creation & Teardown: Gracefully opening a communication channel, authenticating users, assigning session IDs, and tearing down the channel when done.
  • Synchronization & Checkpointing: Injecting checkpoints (recovery markers) into a data stream. If a 100MB download is interrupted at 80MB, session checkpoints allow the transfer to resume from the last marker instead of restarting from scratch.
  • Dialogue Control: Coordinating half-duplex (alternating sender/receiver) or full-duplex (simultaneous) transmission streams so packets don't overlap in confusing sequences.

Analogy: The Courier Clerk & Order Authorization

Amazon Analogy

The Session Layer is like the clerk checking your ID and payment authorization before shipping or releasing the package. They make sure you have an active account (session token) and coordinate the transaction, ensuring the handoff happens only after authentication is confirmed.

Dialogue Modes & Session Lifecycle

Pros
  • Maintains state across stateless lower-layer protocols.
  • Ensures session isolation and security via token management.
  • Protects large transfers from network dropouts via checkpoint markers.
Cons
  • Adds state management overhead on servers (scaling stateful sessions is complex).
  • Sessions can hang or timeout incorrectly if connection drops are not handled gracefully.
Models · OSI Layer 4

Layer 4 — Transport Layer

The engine room, managing segment assembly, error correction, and application sockets.

Definition

The Transport Layer is responsible for host-to-host communication. It takes data from the session layer, breaks it into smaller units called segments, and ensures they are delivered reliably, in the correct order, without error, and directly to the correct application socket using port numbers.

PDU: Segment (TCP) / Datagram (UDP)
Focus: End-to-End Reliability
Hardware: OS Kernel / Sockets

Core Purpose & Function

While the Network Layer below routes packets to the correct *physical computer*, the Transport Layer is the one that gets the data to the correct *software process* on that computer. It acts as the supervisor of the connection, handling flow control, sequencing, and packet loss recovery.

Key Responsibilities

  • Port Addressing & Multiplexing: Attaching source and destination ports. This allows a laptop to run Slack, Zoom, and a browser simultaneously without their data streams mixing.
  • Segmentation & Reassembly: Dividing data into chunks size-optimized for the MTU (Maximum Transmission Unit) of the network, and re-assembling them in order at the destination.
  • Flow Control: Preventing a fast sender from overwhelming a slow receiver by negotiating window sizes.
  • Error Recovery & ACKs: Using checksums to spot corruption and retransmitting lost packets when timers expire.

Analogy: The Shipment Box labeling and Tracking Numbers

Amazon Analogy

The Transport Layer corresponds to the delivery coordinator packing your items into numbered boxes. They label box 1/3, 2/3, 3/3, and require a signature (ACK) upon delivery. If box 2/3 goes missing on the highway, they re-ship box 2/3. At your door, they reassemble the three boxes in order before handing them to you.

Flow Control & Congestion Control

Pros
  • End-to-end data integrity guarantees (for TCP).
  • Enables multitasking (multiplexing) on a single physical link.
  • Prevents packet loss due to receiver buffer overload.
Cons
  • TCP handshakes add latency before any data is sent.
  • Heavy CPU and memory usage to buffer and track outstanding acknowledgements.
Models · OSI Layer 3

Layer 3 — Network Layer

Where packets are stamped with IP addresses and routed across network borders.

Definition

The Network Layer is the routing engine of the internet. It handles the movement of data units called packets across multiple independent networks. It defines logical addressing (IP addresses) and routes packets from their original source host to their final destination using path-selection algorithms.

PDU: Packet
Focus: Inter-Network Routing
Hardware: Routers / Layer 3 Switches

Core Purpose & Function

The Network Layer is where the boundaries of local networks (LANs) are crossed. While Layer 2 can only talk to immediate neighbors on the same cable or switch, Layer 3 is what allows packets to travel across routers, ISPs, and undersea cables to reach a server on the other side of the globe.

Key Responsibilities

  • Logical Addressing: Wrapping segments in IP packets, stamping them with source and destination IP addresses.
  • Routing & Path Selection: Routers calculate the optimal path through the global mesh using routing tables and algorithms (e.g. OSPF, BGP, RIP).
  • Packet Fragmentation: Splitting large packets into smaller fragments if they exceed the MTU of a router interface, and setting flags to enable reassembly.
  • Congestion Avoidance: Dropping or throttling packets when router queues fill up.

Models · OSI Layer 2

Layer 2 — Data Link Layer

Hop-by-hop delivery, framing packets into physical frames, and resolving MAC addresses.

Definition

The Data Link Layer handles direct node-to-node communication on the same local network. It takes IP packets, wraps them in frames containing physical hardware addresses (MAC addresses), detects errors on the wire, and controls access to the physical transmission medium.

PDU: Frame
Focus: Local Hops & Media Access
Hardware: Switches, Bridges, NICs

Core Purpose & Function

Layer 3 moves packets globally, but Layer 2 is what physically walks the packet across individual wires between immediate neighbors. It bridges the logical network layer with the physical copper/glass wire, using MAC addresses to resolve precisely which network card on a shared cable should open a frame.

Key Responsibilities

  • Framing: Wrapping packets with start/stop indicators and error check sequences (FCS/CRC checksums) to establish where a frame begins and ends.
  • Physical Addressing: Adding source and destination MAC addresses to frames. MAC addresses are 48-bit numbers burned into NIC hardware at the factory.
  • Media Access Control (MAC): Coordinating who is allowed to transmit on shared networks (like Wi-Fi frequencies or shared Ethernet hubs) using collision detection or avoidance systems (CSMA/CD, CSMA/CA).
  • Address Resolution (ARP): Finding the hardware MAC address of a destination IP on the local subnet using local ARP broadcasts.

Models · OSI Layer 1

Layer 1 — Physical Layer

Where raw bits travel as copper voltages, light pulses, or radio signals.

Definition

The Physical Layer is the hardware base of the network. It deals with the transmission and reception of unstructured raw data streams (bits) over physical media. It defines electrical voltages, light frequencies, connector shapes, cable specs, and transmission speeds.

PDU: Bit
Focus: Electrical/Optical Signals
Hardware: Hubs, Repeaters, Cables, Modems

Core Purpose & Function

Layer 1 has no understanding of packet headers, addresses, or applications. It is strictly concerned with sending a series of electrical or optical pulses representing "1" or "0" down a wire, and ensuring that a receiver interprets those physical pulses back into matching bits.

Key Responsibilities

  • Bit Representation: Defining how 1s and 0s are coded (e.g. changes in voltage levels, light brightness, or radio frequency phases).
  • Bandwidth & Data Rates: Governing the maximum bit rate (transmission speed in Mbps/Gbps) supported by a cable type.
  • Physical Media Specifications: Standardizing wire thicknesses, fiber core widths, connector shapes (like RJ45 plugs), and shielding specifications.
  • Line Configuration & Duplexing: Defining whether transmission is point-to-point, simplex (one-way), half-duplex, or full-duplex.

Deep Dives · Application Layer

HTTP & the Web

The protocol underneath every webpage you've ever loaded.

Definition

HTTP (HyperText Transfer Protocol) is an application-layer protocol that defines how a client requests a resource from a server, and how the server formats its response. HTTP itself runs on top of TCP — TCP guarantees the underlying bytes actually arrive, HTTP defines what those bytes mean.

Stateless by design

HTTP is a stateless protocol — by default, each request is independent and the server doesn't remember anything about previous requests from the same client. This is exactly why cookies exist: to layer state on top of a fundamentally stateless protocol.

Request & Response headers

Both requests and responses carry metadata in headers — e.g. content type, content length, and Keep-Alive settings, which define a timeout for how long a TCP connection should stay open for reuse before being closed, avoiding the overhead of reconnecting for every single request.

Cookies

Definition

A cookie is a small unique string set by a server and stored in the client's browser, used to maintain state across otherwise-stateless HTTP requests (e.g. staying logged in).

  • The first time you visit a site, the server sets the cookie in your browser.
  • On every subsequent visit, the browser automatically sends that cookie back, letting the server recognize you.
  • Cookies have an expiration date, after which they're discarded.
  • Third-party cookies — cookies set for domains other than the one you're actually visiting (commonly used for cross-site tracking/ads).

Full step-by-step lookup, typing google.com

  1. Your computer first checks its own local cache — if you've visited recently, the IP may already be saved locally.
  2. If not cached, the request goes to your ISP's local DNS server — the first external point of contact.
  3. If the local DNS server doesn't have it cached either, it queries a root server.
  4. The root server doesn't know the IP, but acknowledges and redirects to the correct TLD server (e.g. the .com server).
  5. The TLD server returns the IP address of google.com's actual name server.
  6. Your computer finally connects directly to google.com's server using the resolved IP.

Deep Dives · Transport Layer

TCP vs UDP — Deep Dive

The two transport protocols, and why an application picks one over the other.

Multiplexing & demultiplexing

Multiplexing

Allows many different streams (a message, a file, a video call) to all be sent to the destination through one shared medium/connection, by the Transport Layer combining them with distinct port numbers.

Demultiplexing

The reverse process on the receiving end — using sockets (the gateway between an application and the network, identified by IP + port) to correctly route incoming data back to the right application.

Checksums — how corruption is detected

Both TCP and UDP use checksums to detect data corruption:

  1. The sender computes a checksum value from the data using a defined algorithm and attaches it alongside the data.
  2. The data + checksum travel together across the network.
  3. The receiver re-runs the same algorithm on the received data and compares the result against the attached checksum.
  4. If they match, the data is assumed intact; if not, it's flagged as corrupted.

Reliability mechanics — timers, ACKs, sequencing

  • When data is sent, a timer starts.
  • An acknowledgement (ACK) from the receiver stops the timer, confirming successful delivery.
  • If no ACK arrives before the timer expires, a retransmission is triggered.
  • Problem: if data was actually received but the ACK itself was lost, naive retransmission would cause duplicate data.
  • Solution: sequence numbers let the receiver recognize and discard duplicates, and reassemble data in the correct order even if packets arrive out of sequence.

UDP — User Datagram Protocol

Definition

UDP is a lightweight, connectionless transport protocol. It sends data without establishing a dedicated connection first, and without guaranteeing delivery, order, or integrity beyond a basic checksum.

  • Data may or may not be delivered — no acknowledgements, no retransmission
  • Data may be received out of order, damaged, or duplicated — UDP doesn't fix any of that itself
  • Still uses a checksum, but doesn't act on a failure beyond detecting it
  • Why use it anyway: it's much faster, because it skips all the handshake/acknowledgement/retransmission overhead

Common uses: video conferencing, online gaming, and DNS lookups — all cases where speed matters more than guaranteed perfect delivery (a slightly garbled video frame is fine; waiting for a perfect retransmitted one isn't).

UDP Packet structure

Total header size: 8 bytes, split into four 2-byte fields:

FieldSize
Source Port No.2 bytes
Destination Port No.2 bytes
Length of datagram2 bytes
Checksum2 bytes
Data (payload)variable

4 fields × 2 bytes = 8 bytes header (16 bits each × 4 = 64 bits = 8 bytes)

Try it

Run tcpdump -c 5 in a terminal (with sufficient permissions) to capture and inspect the first 5 live packets crossing your network interface.

TCP — Transmission Control Protocol

Definition

TCP is a connection-oriented, reliable transport protocol. Before any data is sent, both sides establish a dedicated connection, and TCP guarantees that data arrives complete, uncorrupted, and in the correct order.

  • Application layer hands TCP large amounts of raw data; TCP divides it into chunks, adds headers and a checksum to each
  • Tracks two critical things: data that doesn't arrive (via timers/retransmission) and data order (via sequence numbers)
  • Congestion control — dynamically adjusts how much data is sent based on network conditions, to avoid overwhelming the network

The Three-Way Handshake

Before any actual data flows, TCP establishes a connection using a three-step exchange:

Client → SYN (seq=x)
→
Server
Server → SYN-ACK (seq=y, ack=x+1)
→
Client
Client → ACK (seq=x+1, ack=y+1)
→
Server
  1. SYN — client sends a synchronization flag with an initial sequence number, requesting a connection.
  2. SYN-ACK — server acknowledges and sends its own sequence number back (a mathematically derived next value).
  3. ACK — client acknowledges the server's sequence number, and the connection is now fully established — data can now flow bidirectionally.

Deep Dives · Network Layer

IP Addressing & Subnetting

How packets get routed, classed, and divided across the global network.

Routing fundamentals

Every router maintains a routing table (a data structure mapping destination addresses to the next hop) — when computer A sends data to computer B, it doesn't go directly; it's relayed hop-by-hop through a chain of routers, each forwarding it one step closer using its own table, until it reaches the destination.

In graph terms: routers are nodes, the links between them are edges — and finding the best path through this graph is a routing/shortest-path problem.

Static routing

Routes are configured manually by an administrator — simple but doesn't adapt to network changes.

Dynamic routing

Routes are computed and updated automatically as the network changes, using algorithms like Bellman-Ford or Dijkstra's algorithm.

The component that builds and maintains these routing tables is referred to as the control plane of the network.

IPv4 vs IPv6

FeatureIPv4IPv6

Address space2³² ≈ 4.3 billion unique addresses2¹²⁸ ≈ 3.403 × 10³⁸ unique addresses
Adoption challenge—Not backward compatible with IPv4; requires significant ISP/hardware migration

An IPv4 address like 192.168.2.30 splits into a network/subnet ID (e.g. 192.168.2) identifying which network the device belongs to, and a device/host address (the final 30) identifying the specific machine within that network.

Subnet masking

Definition

A subnet mask "masks" the network portion of an IP address, leaving the remaining bits to identify the specific host. It tells devices which part of an address is the network and which part is the individual device.

For a Class C address, the default mask is 255.255.255.0 — the first three octets identify the network, the last identifies the host. CIDR notation (e.g. /24) is the modern compact way to express this, where the number indicates how many leading bits are the network portion.

Worked example — variable length subnetting

Given 192.0.1.0/24: the network covers addresses from 192.0.1.0 (start) to 192.0.1.255 (end) — 256 total addresses, with the /24 meaning the first 24 bits are fixed as the network portion, leaving 8 bits (2⁸ = 256) for host addresses within it.

Reserved & special addresses

  • 127.0.0.0/8 — the loopback range. 127.0.0.1 specifically is "localhost" — it lets a device contact itself, useful for a machine acting as both client and server during local development/testing.

TTL — Time To Live

A counter that decreases by 1 at every router hop. It exists because packets can sometimes get stuck looping between routers — once TTL hits 0, the packet is dropped, preventing it from circulating forever. A typical starting TTL value is around 60–64.

Middleboxes — Firewalls

A firewall is an extra device sitting between your trusted internal network and the broader, untrusted internet, filtering traffic based on configurable rules.

  • Filters by source/destination address
  • Filters by port number
  • Filters by protocol and packet flags
Stateless firewall

Examines each packet in isolation against static rules — faster but less context-aware.

Stateful firewall

Tracks the state of active connections, allowing traffic that's part of a connection it already recognizes as legitimate (e.g. a response to a request you sent) — more secure but more resource-intensive.

Deep Dives · Network Layer

NAT & Private Addressing

How millions of home devices share a single public IP address.

Definition

NAT (Network Address Translation) is a technique that lets multiple devices on a private network share a single public IP address, by having the router rewrite source/destination addresses as packets cross between the private network and the public internet. This dramatically reduces global public IP address consumption.

Devices on a home/office network are typically assigned private IP addresses (e.g. the 192.168.x.x range), which only have meaning inside that local network — the wider internet has no idea those addresses exist.

Worked example — Host ↔ Server through NAT

A host at 10.0.0.1 on a private network wants to reach a server at 200.100.10.1 on the internet, via a router whose public-facing IP is 150.150.0.1:

HopSource IPDestination IP
Host → Router10.0.0.1200.100.10.1
Router → Internet (after NAT)150.150.0.1 (rewritten)200.100.10.1

The server only ever sees 150.150.0.1 as the source — it has no visibility into the private 10.0.0.1 address behind the router. On the way back, the router reverses the translation, rewriting the destination from 150.150.0.1 back to 10.0.0.1 so the reply reaches the correct internal host.

Two devices, one router — subnetting in practice

Device A at 192.168.1.1 wants to reach facebook.com, which resolves to 192.168.7.1 on a network with subnet range 192.168.1.1 – 192.168.2.1. Device B sits at 192.168.2.1 on a separate subnet. The router/gateway in the middle is responsible for knowing which subnet a destination belongs to and routing accordingly — exactly the role subnet masks play in deciding "is this destination local, or does it need to be routed elsewhere."

Why private IP ranges exist

Because IPv4 only offers ~4.3 billion addresses total — far fewer than the number of internet-connected devices worldwide — reserved private ranges (like 192.168.x.x, 10.x.x.x) can be reused identically inside millions of separate home/office networks without conflict, since NAT means only the router's single public IP needs to be globally unique.

Definition

MAC Addresses

Definition

A MAC address (Media Access Control address) is a hardware identifier permanently assigned to a network interface — every network-capable device has one, distinct from its (changeable) IP address.

Within a LAN, devices at the Data Link layer communicate using these MAC addresses, not IP addresses.

ARP — Address Resolution Protocol

Definition

ARP bridges the gap between Layer 3 and Layer 2 — it resolves a known IP address into the corresponding MAC address needed to actually deliver a frame on the local network. Resolved mappings are cached locally in what's called the ARP cache, avoiding repeated lookups for the same address.

Why hubs lost to switches

Although not explicitly named in the original notes, the practical successor to the hub is the switch — it behaves like a smarter hub that, similar to a bridge, reads MAC addresses and forwards data only to the specific port the destination device is on, rather than broadcasting to everyone. This is why switches replaced hubs in virtually all modern wired networks.
