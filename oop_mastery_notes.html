Object-Oriented Programming

A programming paradigm based on the concept of objects that contain data (attributes) and behavior (methods). Built on four pillars.

Class

A blueprint or template that defines properties and behaviors. Like a house plan — it describes structure but isn't a house itself.

Object

An instance of a class. The actual house built from the plan. Has its own state. One class can produce many objects.

The 4 Pillars

Encapsulation

Bundle data + methods in one class. Hide sensitive data with access modifiers.

Inheritance

Child class inherits properties and methods from parent class. Enables code reuse.

Polymorphism

Same interface, different behavior. One function name, many implementations.

Abstraction

Show only what's necessary. Hide complexity. Use abstract classes and interfaces.

Foundation
Classes & Objects

A class is a blueprint. An object is a real instance of that blueprint with its own state.

Class — a user-defined data type that groups together data members (attributes) and member functions (methods) that operate on that data. Think of it as a cookie cutter.

Object — a specific instance of a class. If Teacher is the class, then t1 is an object with name="Rahul", subject="C++", salary=25000.
C++ example
class Teacher {
  public:
    string name;
    string subject;
    int salary;

    void display() {
      cout << name << " teaches " << subject;
    }
};

// Creating objects
Teacher t1; // t1 is an object
t1.name = "Rahul"; // dot operator
t1.subject = "C++";
t1.display();

Dot Operator (.)

Used to access members of an object. t1.name reads the name member from object t1. The bridge between object and its data.

Memory Allocation

Memory is allocated when the object is created (when the constructor is called). Each object gets its own copy of data members. Static members are shared.

Encapsulation
Access Modifiers

Control who can see and use which data members and methods. The primary tool for data hiding.

public

Accessible from anywhere — inside the class, outside the class, in derived classes. Least restrictive.

private

Accessible only inside the class itself. Not even derived classes can access. Most restrictive. Default in C++ classes.

protected

Accessible inside the class AND in derived classes. Not accessible from outside. Middle ground.

Inheritance access table (from your image)
Base class Derived: private mode Derived: protected mode Derived: public mode
privateNot inheritedNot inheritedNot inherited
protected→ private→ protected→ protected
public→ private→ protected→ public
Key rule: Private members are NEVER inherited regardless of inheritance mode. Protected and public members are inherited, but their visibility in the derived class is determined by the inheritance mode — it can only become more restrictive, never less.
Getters and Setters (Access Specifiers)
Since private data can't be accessed directly from outside, we use getter functions (to read private data) and setter functions (to write private data with validation). This gives controlled access.
class Student {
  private:
    int age;
  public:
    void setAge(int a) { // setter — validates before setting
      if(a > 0) age = a;
    }
    int getAge() { // getter — read-only access
      return age;
    }
};
Pillar 1
Encapsulation

Wrapping data and member functions together in a single unit (class). Like a capsule that hides the medicine inside.

Encapsulation = bundling data (attributes) + methods (behaviors) that operate on that data into a single unit called a class. The internal implementation is hidden — users only interact through the public interface.

Data hiding is achieved by marking sensitive members as private. External code cannot accidentally or maliciously corrupt the internal state.

The Capsule Analogy

A medicine capsule hides its complex chemical formula inside. You just swallow it — you don't need to know the internals. Similarly, a class exposes a clean public interface and hides implementation details.

Why Encapsulate?

Protects data integrity (validation in setters), reduces coupling between components, easier to change internals without breaking external code, and makes debugging simpler.

// Bad — no encapsulation
Student s; s.age = -5; // invalid, but allowed

// Good — encapsulated
s.setAge(-5); // setter rejects invalid value
s.setAge(20); // valid, stored safely
Object Lifecycle
Constructors

A special method invoked automatically when an object is created. Used for initialization. Never has a return type.

Constructor — a special member function with the same name as the class. Automatically called at object creation. No return type (not even void). Used to initialize data members. Memory is allocated when the constructor runs.
Types of constructors

Default (non-parameterized) constructor

No parameters. Gives default values. Called automatically when you write Teacher t1;

Teacher() {
  name = "Unknown"; salary = 0;
}

Parameterized constructor

Accepts arguments to initialize with specific values. You pass values at object creation.

Teacher(string n, int s) {
  name = n; salary = s;
}
// Usage:
Teacher t1("Rahul", 25000);

Copy constructor

Creates a new object as a copy of an existing one. Passed by reference to avoid infinite recursion.

Teacher(Teacher &orig) {
  name = orig.name;
  salary = orig.salary;
}
// Usage: Teacher t2(t1); — copies t1 into t2
Constructor overloading
Constructor overloading — multiple constructors in the same class, each with different parameters. C++ picks the right one based on what arguments you pass. This is compile-time polymorphism.
Shallow vs Deep Copy
Shallow copy
  • Copies all member values as-is
  • Pointers are copied — both objects point to the same memory
  • Default copy constructor does this
  • Dangerous with dynamic memory — double-free crash
Deep copy
  • Copies values AND allocates new memory
  • Pointers point to separate independent copies
  • Must write custom copy constructor
  • Safe — no shared memory between objects
// Deep copy — allocate new memory
Teacher(Teacher &orig) {
  name = orig.name;
  data = new int[size]; // new allocation!
  copy(orig.data, orig.data+size, data);
}
Object Lifecycle
Destructors

The opposite of a constructor. Called automatically when an object goes out of scope to free allocated memory.

Destructor — a special member function prefixed with ~ that runs automatically when an object is destroyed (goes out of scope or is explicitly deleted). Its job: release any dynamic memory the object owns to prevent memory leaks.

Constructor → Destructor order

Objects are constructed in order. Destroyed in reverse order (LIFO). In inheritance: base constructor → derived constructor → ... → derived destructor → base destructor.

delete keyword

delete ptr — frees heap memory for a single object.
delete[] arr — frees heap memory for an array.
Always pair new with delete to avoid leaks.

class Teacher {
  int* data;
  public:
    Teacher() { data = new int[100]; } // alloc
    ~Teacher() { delete[] data; } // free — ~symbol
};

// Manual delete for heap objects:
Teacher* t = new Teacher();
delete t; // calls destructor + frees memory
Static objects live for the entire lifetime of the program. Their destructor is called only when the program exits — not when a function returns.
Memory
The this Pointer

A special hidden pointer in every member function that points to the object that called the function.

this is an implicit pointer available inside every non-static member function. It holds the address of the current object. Used to disambiguate when a local variable shadows a member variable, or to return the current object from a method.
class Teacher {
  string name;
  public:
    void setName(string name) {
      // 'name' (param) shadows 'name' (member)
      this->name = name; // this->name = member
    }
};

// this->prop is the same as (*this).prop
// Used in copy constructor to avoid ambiguity

this->prop

Accesses the current object's member. Equivalent to (*this).prop. Arrow operator dereferences the pointer and accesses the member in one step.

Return *this

Return the current object by reference to enable method chaining: obj.setName("A").setSalary(100). Each method returns *this (the object itself).

Pillar 2
Inheritance

A derived class acquires properties and behavior from a base class. Enables code reuse — write once, inherit everywhere.

Inheritance — when a child (derived) class inherits the data members and member functions of a parent (base) class. The child can use, extend, or override inherited behavior without rewriting it.

Types of inheritance

1. Single inheritance

One parent, one child. Simplest form.

Person
Student
class Student : public Person { };

2. Multi-level inheritance

Chain of inheritance: grandparent → parent → child.

Person
Teacher
MathTeacher

3. Multiple inheritance

One child, multiple parents.

Student
Employee
Intern
class Intern : public Student, public Employee { };

4. Hierarchical inheritance

One parent, multiple children.

Person
Student
Teacher

5. Hybrid inheritance

Mix of multiple inheritance types. Example: Person → Student + Teacher → TA.

Hybrid with multiple inheritance can create the Diamond Problem — ambiguity about which base class member to use. C++ solves this with virtual base classes.
Pillar 3
Polymorphism

"Many forms." Same function name, different behavior depending on context. Two kinds: compile-time and runtime.

Polymorphism — the ability of objects to take on different forms or behave differently depending on context. One interface, multiple implementations.
Compile-time (static)
Run-time (dynamic)
Compile-time polymorphism — the compiler resolves which function to call at compile time. No runtime overhead. Two forms: function overloading and operator overloading.

Function overloading

Same function name, different parameter list (count or type). Compiler picks the right version based on arguments.

int add(int a, int b) { return a+b; }
float add(float a, float b) { return a+b; }
int add(int a, int b, int c) { return a+b+c; }

add(1, 2); // calls version 1
add(1.1f, 2.2f); // calls version 2
add(1, 2, 3); // calls version 3

Operator overloading

Redefine how standard operators (+, -, ==) work with custom objects.

class Complex {
  int real, imag;
  Complex operator+(Complex c2) {
    Complex res;
    res.real = real + c2.real;
    return res;
  }
};
Complex c3 = c1 + c2; // calls operator+
Run-time polymorphism — function to call is decided at runtime via virtual function tables. Requires inheritance. Enables flexible, extensible designs.

Function overriding

Parent and child both define the same function. Child's version replaces parent's. Depends on inheritance.

class Animal {
  virtual void speak() { cout << "..."; }
};
class Dog : public Animal {
  void speak() { cout << "Woof!"; } // overrides
};
Animal* a = new Dog();
a->speak(); // prints "Woof!" — runtime dispatch

Virtual functions

Declared with virtual in the base class. C++ uses a vtable (virtual function table) to resolve the correct override at runtime. Must be overridden in derived class.

virtual void draw() = 0; // pure virtual — must override
// Class with pure virtual = abstract class
// Cannot instantiate abstract class directly
Without virtual, the base class version is always called regardless of the actual object type (static binding). With virtual, the correct override is called at runtime (dynamic binding).
Pillar 4
Abstraction

Hide unnecessary implementation details. Expose only what the user of your class needs to know.

Abstraction — showing only the essential features while hiding the complex implementation. You drive a car without knowing how the engine works. You use sort() without knowing the algorithm. That's abstraction.

vs. Encapsulation

Encapsulation is the mechanism (bundling + private). Abstraction is the design principle (hiding complexity). Encapsulation implements abstraction.

Abstraction: what to hide. Encapsulation: how to hide it.

Abstract classes

A class with at least one pure virtual function. Cannot be instantiated directly. Meant to be a base class — defines the interface that derived classes must implement.

class Shape { // abstract class
  public:
    virtual float area() = 0; // pure virtual
};

class Circle : public Shape {
  float r;
  float area() { return 3.14f * r * r; } // must implement
};

// Shape s; // ERROR — cannot instantiate abstract class
Shape* s = new Circle(); // OK — pointer to abstract
Memory
Static Keyword

Static variables exist for the entire program lifetime. Static class members are shared across all objects.

Static local variable

Declared inside a function. Created and initialized only once. Retains its value between function calls. Lives until program ends.

void counter() {
  static int count = 0; // init once
  count++;
  cout << count; // 1, 2, 3 on each call
}

Static class member

Belongs to the class, not any specific object. Shared by all objects. Must be initialized outside the class definition. Accessed via ClassName::member.

class Bank {
  static float rate; // shared
};
float Bank::rate = 5.5f;
// Bank::rate — same for all objects
Static objects — created once, live for the entire program lifetime. Destructor called only when the program terminates. Stored in a special memory segment (BSS/data), not on the stack.
Special
Friend Functions & Classes

Grant specific external functions or classes access to private and protected members — without making them part of the class.

A friend function is declared inside a class with the friend keyword. It's not a member of the class, but it can access private and protected members. Friendship is granted by the class, not claimed by the function.

Friend function

An external function that can access private members of a class. Declared inside the class with friend keyword. Defined outside the class like a regular function.

class Box {
  private: int volume;
  friend void printVolume(Box b);
};
void printVolume(Box b) {
  cout << b.volume; // can access private
}

Friend class

An entire class is granted access to another class's private members. Used when two closely related classes need to work intimately together (e.g., iterator + container).

class Engine;
class Car {
  friend class Engine; // Engine can see Car's private
  private: int fuelLevel;
};
Overusing friend breaks encapsulation. Use sparingly — only when the external function genuinely needs deep access and can't get it through public methods.

Quick Reference
Deep vs Shallow Copy

Copying objects that contain pointers requires care. Shallow copy shares memory — dangerous. Deep copy duplicates it — safe.

Shallow copy

Copies member values bit-by-bit. Pointer members are copied — both objects point to the same heap memory.

int* ptr = new int(42);
int* copy = ptr; // same address!
delete ptr; // deletes the data
// copy now points to freed memory — crash!

Deep copy

Allocates new memory and copies the data into it. Both objects are fully independent.

int* ptr = new int(42);
int* copy = new int(*ptr); // new alloc
delete ptr; // safe — copy is independent
Rule of Three (C++): If you define any of destructor, copy constructor, or copy assignment operator — you should define all three. This ensures deep copies everywhere dynamic memory is used.
When does this matter? Only when your class owns heap memory (via new). If all members are value types (int, string, etc.), the default copy constructor is fine.
